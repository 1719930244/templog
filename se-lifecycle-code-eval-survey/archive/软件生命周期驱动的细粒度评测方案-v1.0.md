# 软件生命周期驱动的 AI 代码生成细粒度评测方案 v1.0
## 总览
以软件生命周期的流程（需求设计、软件构造、测试验证、维护演进）进行维度划分与逻辑合并。方案在设计上重点考虑了以下三个维度：数据复用性与一致性：通过在不同阶段复用同一数据集），追踪模型从架构设计到具体实现的一致性表现，易于实操。时效性与有效性：考虑已有SOTA模型在测评集的表现，保证评测结果有价值导向。细粒度指标拆分：不同于仅关注如“Issue 是否修复、功能是否实现”的结果导向，本方案引入指令遵循率 (IFR)、模块内聚度 (LCOM)、变异测试得分等多种细粒度指标，深度剖析模型在软件工程各环节的“中间过程”质量，从而区分AI模型生成代码的细粒度评测。

## 核心思路与划分依据

### 为什么按软件生命周期 4 阶段划分？

**传统评测的局限**：
- 现有 AI 代码生成评测（如 HumanEval, MBPP）主要关注"代码是否通过测试"（pass@k），忽略了软件工程的全生命周期
- 单一的 pass@k 指标无法区分"差一点通过"和"完全错误"，也不衡量代码质量、可维护性、安全性等非功能属性
- 真实软件开发不仅是"写代码"，还包括需求理解、架构设计、测试生成、Bug 修复、代码审查等多个阶段

**本方案的设计理念**：
1. **阶段 1（需求与设计）**：评估 AI 对需求的理解和转化能力
   - **为什么重要**：需求理解错误会导致整个实现方向错误，是软件失败的首要原因
   - **评测重点**：指令遵循、API 选择、架构设计、领域需求理解
   - **数据集选择**：BigCodeBench（复杂 API 调用 + 约束条件）、ProjDevBench（项目级架构）、EvoCodeBench（领域特定需求）

2. **阶段 2（软件构造）**：评估生成代码的质量和效率
   - **为什么重要**：代码质量直接影响可维护性和长期成本，效率影响用户体验
   - **评测重点**：代码质量（异味、复杂度）、执行效率（时间、内存）、API 调用正确性
   - **数据集复用**：ProjDevBench（代码质量）、EvoCodeBench（效率）、BigCodeBench（API 正确性）

3. **阶段 3（测试）**：评估 AI 生成测试用例的能力
   - **为什么重要**：测试是保证代码质量的关键，AI 辅助测试生成可大幅提升开发效率
   - **评测重点**：测试生成质量、覆盖率、变异测试得分
   - **数据集选择**：TestGenEval（单元测试生成）、EvoCodeBench（测试覆盖率）

4. **阶段 4（维护）**：评估 AI 在 Bug 修复和代码审查中的能力
   - **为什么重要**：软件维护占软件生命周期成本的 60-80%，AI 辅助维护潜力巨大
   - **评测重点**：Bug 定位与修复、代码审查质量
   - **数据集选择**：RepoDebug（仓库级调试）、AACR-Bench（PR 级代码审查）

## 阶段 1：需求与设计

### 数据集组合
- **BigCodeBench**（指令遵循与 API 设计）
- **ProjDevBench**（架构设计与模块划分）
- **EvoCodeBench**（领域需求理解）

### 1.1 BigCodeBench：指令遵循与 API 设计

**典型示例**：
- 任务：实现数据分析管道，需调用 `pandas`, `numpy`, `matplotlib` 等库
- 输入：自然语言需求描述 + 139 个可用库的文档 + **约束条件**（如"不使用 for 循环"、"函数签名必须是 def analyze(data: pd.DataFrame) -> dict"）
- 预期输出：包含多个 API 调用的函数实现（平均 5-7 个函数调用/任务）

**评测维度**：
1. **指令遵循率 (IFR)**（核心指标）
   - 硬约束：AST 结构、导入限制、禁用 API（可自动验证）
   - 运行时约束：行为、性能、输出格式（需执行验证）
   - 软约束：代码风格、命名偏好（需 LLM 判定）

2. **API 选择合理性**（设计层面）
   - 是否选择了最适合需求的库（如用 `pandas.DataFrame` 而非手动循环）
   - 是否避免了冗余依赖

3. **库依赖设计**
   - 依赖数量是否合理
   - 是否存在循环依赖风险

**指标计算**：

```
指令遵循率 (IFR)：

IFR_hard    = 通过的硬约束数 / 硬约束总数
IFR_runtime = 通过的运行时约束数 / 运行时约束总数
IFR_soft    = 通过的软约束数 / 软约束总数

IFR = (IFR_hard × n_hard + IFR_runtime × n_runtime + IFR_soft × n_soft) / (n_hard + n_runtime + n_soft)

其中：
- 硬约束：函数签名、禁用 API、导入限制等（tree-sitter AST 检查）
- 运行时约束：输出格式、性能要求等（执行验证）
- 软约束：代码风格、命名规范等（LLM-as-Judge）
- 当某类约束为空时，该类不计入分母

API 选择合理性得分 = (正确 API 数 / 总 API 调用数) × 100%

其中：
- 正确 API：专家标注的最优 API 选择
- 总 API 调用数：生成代码中的所有 API 调用

依赖复杂度 = Σ(库 i 的依赖深度) / 使用的库总数

其中：
- 依赖深度：该库在依赖树中的层级（直接依赖=1，间接依赖=2，...）
- 越低越好（理想值 ≤ 1.5）
```

### 1.2 ProjDevBench：架构设计与模块划分

**典型示例**：
- 仓库：`flask-restful` 项目（RESTful API 框架）
- 需求：设计一个支持资源路由、请求解析、响应序列化的架构
- 预期输出：多文件项目结构（5-10 个模块文件）

**评测维度**：
1. **模块划分合理性**
   - 是否遵循单一职责原则
   - 模块间耦合度

2. **设计异味检测**
   - Cyclic Dependency（循环依赖）
   - God Class（上帝类）
   - Feature Envy（特性依恋）

**指标计算**：

```
模块内聚度 (LCOM) = 1 - (Σ 方法对共享属性数 / (方法数 × 属性数))

其中：
- 方法对：模块内所有方法的两两组合
- 共享属性数：每对方法共同访问的属性数量
- 理想值：LCOM ≤ 0.3（高内聚）

模块耦合度 (CBO) = 该模块引用的外部类数 + 引用该模块的外部类数

其中：
- 外部类：不在当前模块内的类
- 理想值：CBO ≤ 5

设计异味得分 = 1 - (检测到的设计异味数 / 总模块数)

其中：
- 设计异味：使用 Designite 工具检测的架构级异味
- 包括：Cyclic Dependency, God Class, Feature Envy 等
```

### 1.3 EvoCodeBench：领域需求理解

**典型示例**：
- 领域：Database（数据库操作）
- 任务：实现一个支持事务、索引、查询优化的 ORM 层
- 10 个领域：Database, Internet, GUI, Algorithm, Data Science, Security, System, Testing, DevOps, Multimedia

**评测维度**：
1. **领域特定需求覆盖**
   - 是否实现了领域核心功能（如数据库的 ACID 特性）
   - 是否使用了领域特定的设计模式（如 ORM 的 Active Record 模式）

2. **算法设计选择**
   - 是否选择了适合该领域的算法（如数据库索引用 B+ 树而非哈希表）

**指标计算**：

```
领域需求覆盖率 = (实现的领域核心功能数 / 领域核心功能总数) × 100%

其中：
- 领域核心功能：EvoCodeBench 为每个领域定义的必需功能列表
- 例如 Database 领域：CRUD, Transaction, Index, Query Optimization

领域特定改进 (DSI) = (模型在该领域的 Pass@1 - 模型在所有领域的平均 Pass@1) / 模型在所有领域的平均 Pass@1

其中：
- DSI > 0：该领域是模型的"舒适区"
- DSI < 0：该领域是模型的"陌生区"
```

## 阶段 2：软件构造

### 数据集组合
- **ProjDevBench**（代码质量与实现异味）
- **EvoCodeBench**（代码效率与资源消耗）
- **BigCodeBench**（API 调用正确性）

### 2.1 ProjDevBench：代码质量与实现异味

**典型示例**（复用阶段 1 的 `flask-restful` 项目）：
- 架构设计已完成（阶段 1 输出）
- 现在评估：具体实现的代码质量
- 生成代码量级：500-1000 行 Python 代码，分布在 5-10 个文件

**评测维度**：
1. **细粒度代码属性**
   - 可读性：命名规范、注释质量
   - 可维护性：函数长度、圈复杂度
   - 健壮性：异常处理、边界条件

2. **实现异味检测**
   - Long Method（长方法）
   - Magic Number（魔法数字）
   - Duplicate Code（重复代码）

**指标计算**：

```
圈复杂度 (CC) = E - N + 2P

其中：
- E：控制流图的边数
- N：控制流图的节点数
- P：连通分量数（通常为 1）
- 理想值：CC ≤ 10

代码重复率 = (重复代码行数 / 总代码行数) × 100%

其中：
- 重复代码：连续 6 行以上的相同代码块
- 理想值：≤ 5%

实现异味密度 = 检测到的实现异味数 / KLOC (千行代码)

其中：
- 实现异味：使用 Pylint/Flake8 检测的代码级异味
- 包括：Long Method (>50 行), Magic Number, Duplicate Code 等
- 理想值：≤ 5 个/KLOC
```

### 2.2 EvoCodeBench：代码效率与资源消耗

**典型示例**（复用阶段 1 的 Database 领域任务）：
- 任务：实现 B+ 树索引的插入和查询
- 输入：10,000 条数据插入 + 1,000 次查询
- 评估：执行时间、内存消耗

**评测维度**：
1. **时间效率**
   - 算法时间复杂度是否最优
   - 实际执行时间

2. **空间效率**
   - 内存消耗峰值
   - 是否存在内存泄漏

**指标计算**：

```
时间效率得分 = 1 - (实际执行时间 / 超时阈值)

其中：
- 超时阈值：EvoCodeBench 为每个任务设定的时间限制（通常 10-60 秒）
- 得分范围：[0, 1]，越高越好

内存效率得分 = 1 - (峰值内存消耗 / 内存限制)

其中：
- 内存限制：通常为 2GB
- 得分范围：[0, 1]，越高越好

算法复杂度匹配度 = (实际复杂度 == 理论最优复杂度) ? 1 : 0

其中：
- 实际复杂度：通过代码静态分析得出
- 理论最优复杂度：该问题的已知最优解（如 B+ 树查询为 O(log n)）
```

### 2.3 BigCodeBench：API 调用正确性

**典型示例**（复用阶段 1 的数据分析任务）：
- 任务：使用 `pandas` 处理 CSV 数据并生成可视化
- API 调用序列：`pd.read_csv()` → `df.groupby()` → `df.agg()` → `plt.plot()`
- 评估：每个 API 调用是否正确、参数是否合法

**评测维度**：
1. **API 调用正确性**
   - 参数类型是否匹配
   - 返回值是否正确处理

2. **错误处理**
   - 是否捕获了 API 可能抛出的异常
   - 异常处理是否合理

**指标计算**：

```
API 调用正确率 = (正确的 API 调用数 / 总 API 调用数) × 100%

其中：
- 正确的 API 调用：参数类型正确 + 返回值正确处理 + 无运行时错误
- 通过单元测试验证

异常处理覆盖率 = (有异常处理的 API 调用数 / 可能抛出异常的 API 调用数) × 100%

其中：
- 可能抛出异常的 API：根据 API 文档标注
- 异常处理：try-except 块或错误码检查
```

## 阶段 3：测试

### 数据集组合
- **TestGenEval**（单元测试生成质量）
- **EvoCodeBench**（测试覆盖率评估）

### 3.1 TestGenEval：单元测试生成质量

**典型示例**：
- 源代码：Python 函数 `calculate_discount(price, coupon_code)`
- 输入：函数签名 + 文档字符串
- 预期输出：完整的 pytest 测试用例（5-10 个测试函数）
- 数据规模：68,647 个测试用例，来自 11 个 Python 仓库

**评测维度**：
1. **测试用例质量**
   - 是否覆盖了正常路径和异常路径
   - 是否包含边界条件测试

2. **测试可执行性**
   - 生成的测试是否能通过语法检查
   - 是否能成功运行

**指标计算**：

```
测试通过率 = (通过的测试用例数 / 生成的测试用例总数) × 100%

其中：
- 通过的测试：语法正确 + 运行无错误 + 断言有效
- 通过 pytest 执行验证

分支覆盖率 = (覆盖的分支数 / 总分支数) × 100%

其中：
- 分支：if-else, switch-case, 循环等控制流分支
- 使用 coverage.py 工具测量

变异测试得分 = (杀死的变异体数 / 总变异体数) × 100%

其中：
- 变异体：对源代码进行微小修改（如 + 改为 -）后的版本
- 杀死：测试用例能检测到该变异（测试失败）
- 使用 mutmut 工具生成变异体
```

### 3.2 EvoCodeBench：测试覆盖率评估

**典型示例**（复用阶段 2 的 Database 领域 B+ 树实现）：
- 源代码：B+ 树的插入、查询、删除函数
- 生成测试：针对这些函数的测试用例
- 评估：测试是否覆盖了所有边界条件（空树、单节点、满节点、分裂、合并等）

**指标计算**：

```
语句覆盖率 = (执行的语句数 / 总语句数) × 100%

路径覆盖率 = (执行的路径数 / 总路径数) × 100%

其中：
- 路径：从函数入口到出口的所有可能执行路径
- 总路径数：通过控制流图计算（可能指数级增长）
- 实际中通常采样关键路径
```

## 阶段 4：维护

### 数据集组合
- **RepoDebug**（Bug 定位与修复）
- **AACR-Bench**（代码审查质量）

### 4.1 RepoDebug：Bug 定位与修复

**典型示例**：
- 仓库：`requests` 库（Python HTTP 客户端）
- Bug 类型：`NullPointerException` 在 `Session.send()` 方法
- 错误信息：堆栈跟踪 + 失败的测试用例
- 涉及文件：3-5 个相关文件（跨文件依赖）
- 数据规模：8 种编程语言，22 种错误类型

**评测维度**：
1. **Bug 定位准确性**
   - 是否定位到正确的文件和行号
   - 是否识别了根本原因（而非表面症状）

2. **修复策略合理性**
   - 修复是否解决了根本问题
   - 是否引入了新的 Bug

3. **跨文件追踪能力**
   - 是否追踪了跨文件的调用链
   - 是否理解了模块间的依赖关系

**指标计算**：

```
Bug 定位准确率 = (正确定位的 Bug 数 / 总 Bug 数) × 100%

其中：
- 正确定位：定位到的文件和行号在专家标注的范围内（±5 行）

修复成功率 = (成功修复的 Bug 数 / 尝试修复的 Bug 数) × 100%

其中：
- 成功修复：修复后所有测试通过 + 未引入新 Bug（通过回归测试验证）

跨文件追踪深度 = Σ(Bug i 涉及的文件数) / 总 Bug 数

其中：
- 涉及的文件数：从 Bug 触发点到根本原因的调用链中的文件数
- 平均值越高，说明模型的跨文件理解能力越强
```

### 4.2 AACR-Bench：代码审查质量

**典型示例**：
- Pull Request：为 `django` 项目添加新的 ORM 查询优化
- 代码变更：修改了 3 个文件，新增 200 行代码
- 审查任务：发现潜在缺陷、性能问题、安全漏洞
- 数据规模：AI 辅助 + 专家验证，发现缺陷数比传统方法多 285%

**评测维度**：
1. **缺陷发现能力**
   - 是否发现了所有专家标注的缺陷
   - 是否发现了专家未标注的新缺陷

2. **审查质量**
   - 审查意见是否具体、可操作
   - 是否提供了修复建议

3. **跨文件上下文理解**
   - 是否理解了代码变更对其他模块的影响
   - 是否发现了跨文件的一致性问题

**指标计算**：

```
缺陷召回率 = (发现的真实缺陷数 / 专家标注的缺陷总数) × 100%

缺陷精确率 = (发现的真实缺陷数 / 报告的缺陷总数) × 100%

其中：
- 真实缺陷：专家确认的缺陷
- 报告的缺陷：模型报告的所有缺陷（包括误报）

F1 得分 = 2 × (精确率 × 召回率) / (精确率 + 召回率)

审查意见质量 = Σ(意见 i 的可操作性得分) / 审查意见总数

其中：
- 可操作性得分：专家评分（1-5 分）
  - 5 分：具体指出问题 + 提供修复建议 + 解释原因
  - 3 分：指出问题但缺少修复建议
  - 1 分：模糊的意见，无法操作
```

## 各阶段综合得分计算

**阶段 1（需求与设计）**：
```

```

**阶段 4（维护）**：
```
维护得分 = 0.4 × Bug 定位准确率 + 0.4 × 修复成功率 + 0.2 × 缺陷 F1 得分
```

### 评测流程

```
1. 数据准备
   ├─ 加载数据集（BigCodeBench, ProjDevBench等）
   ├─ 解析任务描述（需求、约束、测试用例）
   └─ 准备评测环境（依赖安装、沙箱配置）

2. 代码生成
   ├─ 模型推理（Complete/Instruct模式）
   ├─ 代码清洗（去除注释、格式化）
   └─ 语法检查（tree-sitter验证）

3. 阶段1评测（需求与设计）
   ├─ IFR计算（硬约束+运行时约束+软约束）
   ├─ API选择分析（专家标注对比）
   ├─ 架构质量（LCOM, CBO, 设计异味）
   └─ 领域需求覆盖（功能清单对比）

4. 阶段2评测（软件构造）
   ├─ 代码质量（CC, 重复率, 实现异味）
   ├─ 执行效率（时间、内存监控）
   └─ API正确性（单元测试 + 异常处理）

5. 阶段3评测（测试）
   ├─ 测试生成（pytest执行）
   ├─ 覆盖率分析（语句、分支、路径）
   └─ 变异测试（mutmut变异体生成）

6. 阶段4评测（维护）
   ├─ Bug定位（专家标注对比）
   ├─ 修复验证（测试通过 + 回归测试）
   └─ 代码审查（缺陷发现 + 意见质量）

7. 结果汇总
   ├─ 各阶段得分计算
   ├─ 综合得分计算
   └─ 生成评测报告（JSON + 可视化）
```
