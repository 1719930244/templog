# 软件生命周期驱动的 AI 代码生成细粒度评测方案 v1.1

> **版本说明**：v1.1 在 v1.0 基础上补充了需要人工标注和自行计算的高级指标，来源于 v0.4 提案

## 数据集概览

| 数据集 | 发布单位 | 数据规模 | 发布会议/期刊 | 年份 |
|--------|----------|----------|---------------|------|
| **BigCodeBench** | University of Illinois at Urbana-Champaign | 1,140 个函数级任务，涉及 139 个库 | arXiv (待发表) | 2024 |
| **ProjDevBench** | 上海人工智能实验室、香港中文大学等 | 多文件项目级任务 | arXiv 2403.08604 | 2024 |
| **EvoCodeBench** | 清华大学、**阿里巴巴通义实验室**等 | 275 个样本（首版 EvoCodeBench-2403），来自 25 个仓库，10 个编程领域 | NeurIPS 2024 | 2024 |
| **TestGenEval** | Carnegie Mellon University | 68,647 个测试用例，来自 1,210 个代码-测试文件对，覆盖 11 个 Python 仓库 | ICLR 2025 | 2025 |
| **RepoDebug** | Beihang University (北京航空航天大学) | 仓库级调试任务，支持 8 种编程语言，22 种错误类型 | EMNLP 2025 Findings | 2025 |
| **AACR-Bench** | **阿里巴巴 Aone 团队**、多机构联合 | AI 辅助 + 专家验证的代码审查数据，缺陷发现数比传统方法多 285% | arXiv 2601.19494 | 2025 |

**数据集特点**：
- **持续更新**：EvoCodeBench 每 6 个月更新一次，避免数据污染
- **真实场景**：所有数据集均来自真实开源项目（GitHub 等）
- **多语言支持**：RepoDebug 支持 8 种语言，AACR-Bench 支持多语言代码审查
- **跨阶段复用**：同一数据集可在不同生命周期阶段从不同维度评估

## 核心思路

以软件生命周期 4 阶段为主线 → 数据集跨阶段复用 → 具体数据实例 → 详细指标计算公式 → **补充人工标注指标**

## 阶段 1：需求与设计

### 数据集组合
- **BigCodeBench**（API 设计）
- **ProjDevBench**（架构设计）
- **EvoCodeBench**（领域需求）

### 1.1 BigCodeBench：API 选择与库依赖设计

**典型示例**：
- 任务：实现数据分析管道，需调用 `pandas`, `numpy`, `matplotlib` 等库
- 输入：自然语言需求描述 + 139 个可用库的文档
- 预期输出：包含多个 API 调用的函数实现（平均 5-7 个函数调用/任务）

**评测维度**：
1. **API 选择合理性**（设计层面）
   - 是否选择了最适合需求的库（如用 `pandas.DataFrame` 而非手动循环）
   - 是否避免了冗余依赖

2. **库依赖设计**
   - 依赖数量是否合理
   - 是否存在循环依赖风险

**指标计算**：

```
API 选择合理性得分 = (正确 API 数 / 总 API 调用数) × 100%

其中：
- 正确 API：专家标注的最优 API 选择
- 总 API 调用数：生成代码中的所有 API 调用

依赖复杂度 = Σ(库 i 的依赖深度) / 使用的库总数

其中：
- 依赖深度：该库在依赖树中的层级（直接依赖=1，间接依赖=2，...）
- 越低越好（理想值 ≤ 1.5）
```

### 1.2 ProjDevBench：架构设计与模块划分

**典型示例**：
- 仓库：`flask-restful` 项目（RESTful API 框架）
- 需求：设计一个支持资源路由、请求解析、响应序列化的架构
- 预期输出：多文件项目结构（5-10 个模块文件）

**评测维度**：
1. **模块划分合理性**
   - 是否遵循单一职责原则
   - 模块间耦合度

2. **设计异味检测**
   - Cyclic Dependency（循环依赖）
   - God Class（上帝类）
   - Feature Envy（特性依恋）

**指标计算**：

```
模块内聚度 (LCOM) = 1 - (Σ 方法对共享属性数 / (方法数 × 属性数))

其中：
- 方法对：模块内所有方法的两两组合
- 共享属性数：每对方法共同访问的属性数量
- 理想值：LCOM ≤ 0.3（高内聚）

模块耦合度 (CBO) = 该模块引用的外部类数 + 引用该模块的外部类数

其中：
- 外部类：不在当前模块内的类
- 理想值：CBO ≤ 5

设计异味得分 = 1 - (检测到的设计异味数 / 总模块数)

其中：
- 设计异味：使用 Designite 工具检测的架构级异味
- 包括：Cyclic Dependency, God Class, Feature Envy 等
```

### 1.3 EvoCodeBench：领域需求理解

**典型示例**：
- 领域：Database（数据库操作）
- 任务：实现一个支持事务、索引、查询优化的 ORM 层
- 10 个领域：Database, Internet, GUI, Algorithm, Data Science, Security, System, Testing, DevOps, Multimedia

**评测维度**：
1. **领域特定需求覆盖**
   - 是否实现了领域核心功能（如数据库的 ACID 特性）
   - 是否使用了领域特定的设计模式（如 ORM 的 Active Record 模式）

2. **算法设计选择**
   - 是否选择了适合该领域的算法（如数据库索引用 B+ 树而非哈希表）

**指标计算**：

```
领域需求覆盖率 = (实现的领域核心功能数 / 领域核心功能总数) × 100%

其中：
- 领域核心功能：EvoCodeBench 为每个领域定义的必需功能列表
- 例如 Database 领域：CRUD, Transaction, Index, Query Optimization

领域特定改进 (DSI) = (模型在该领域的 Pass@1 - 模型在所有领域的平均 Pass@1) / 模型在所有领域的平均 Pass@1

其中：
- DSI > 0：该领域是模型的"舒适区"
- DSI < 0：该领域是模型的"陌生区"
```

### 1.4 需求歧义处理能力（人工标注指标）

**定义**：当需求描述存在歧义、不完整或矛盾时，模型是否能识别并主动澄清，而非盲目生成。

**典型示例**：
- 需求："实现一个排序函数"（歧义：升序还是降序？稳定排序？）
- 期望行为：模型询问"需要升序还是降序排序？"或"是否需要稳定排序？"
- 数据来源：从 Stack Overflow 的"需要更多信息"类问题中采集

**评测维度**：
1. **歧义识别率**：模型能识别出多少歧义需求
2. **澄清质量**：提出的澄清问题是否切中要害
3. **容错能力**：在无法澄清时，是否给出合理的默认假设

**实现路径**：
- 人工标注驱动：需构建含歧义需求的测试集
- LLM-as-Judge 初筛 + 人工终审
- 人力需求：2-3 人标注团队（约 2-4 周）

**指标计算**：

```
歧义识别率 = (识别出的歧义需求数 / 总歧义需求数) × 100%

澄清质量得分 = Σ(澄清问题 i 的质量评分) / 澄清问题总数

其中：
- 质量评分：专家评分（1-5 分）
  - 5 分：准确识别歧义 + 提出关键澄清问题
  - 3 分：识别歧义但问题不够具体
  - 1 分：未识别歧义或问题无关
```

## 阶段 2：软件构造

### 数据集组合
- **ProjDevBench**（代码质量）
- **EvoCodeBench**（代码效率）
- **BigCodeBench**（API 调用正确性）

### 2.1 ProjDevBench：代码质量与实现异味

**典型示例**（复用阶段 1 的 `flask-restful` 项目）：
- 架构设计已完成（阶段 1 输出）
- 现在评估：具体实现的代码质量
- 生成代码量级：500-1000 行 Python 代码，分布在 5-10 个文件

**评测维度**：
1. **细粒度代码属性**
   - 可读性：命名规范、注释质量
   - 可维护性：函数长度、圈复杂度
   - 健壮性：异常处理、边界条件

2. **实现异味检测**
   - Long Method（长方法）
   - Magic Number（魔法数字）
   - Duplicate Code（重复代码）

**指标计算**：

```
圈复杂度 (CC) = E - N + 2P

其中：
- E：控制流图的边数
- N：控制流图的节点数
- P：连通分量数（通常为 1）
- 理想值：CC ≤ 10

代码重复率 = (重复代码行数 / 总代码行数) × 100%

其中：
- 重复代码：连续 6 行以上的相同代码块
- 理想值：≤ 5%

实现异味密度 = 检测到的实现异味数 / KLOC (千行代码)

其中：
- 实现异味：使用 Pylint/Flake8 检测的代码级异味
- 包括：Long Method (>50 行), Magic Number, Duplicate Code 等
- 理想值：≤ 5 个/KLOC
```

### 2.2 EvoCodeBench：代码效率与资源消耗

**典型示例**（复用阶段 1 的 Database 领域任务）：
- 任务：实现 B+ 树索引的插入和查询
- 输入：10,000 条数据插入 + 1,000 次查询
- 评估：执行时间、内存消耗

**评测维度**：
1. **时间效率**
   - 算法时间复杂度是否最优
   - 实际执行时间

2. **空间效率**
   - 内存消耗峰值
   - 是否存在内存泄漏

**指标计算**：

```
时间效率得分 = 1 - (实际执行时间 / 超时阈值)

其中：
- 超时阈值：EvoCodeBench 为每个任务设定的时间限制（通常 10-60 秒）
- 得分范围：[0, 1]，越高越好

内存效率得分 = 1 - (峰值内存消耗 / 内存限制)

其中：
- 内存限制：通常为 2GB
- 得分范围：[0, 1]，越高越好

算法复杂度匹配度 = (实际复杂度 == 理论最优复杂度) ? 1 : 0

其中：
- 实际复杂度：通过代码静态分析得出
- 理论最优复杂度：该问题的已知最优解（如 B+ 树查询为 O(log n)）
```

### 2.3 BigCodeBench：API 调用正确性

**典型示例**（复用阶段 1 的数据分析任务）：
- 任务：使用 `pandas` 处理 CSV 数据并生成可视化
- API 调用序列：`pd.read_csv()` → `df.groupby()` → `df.agg()` → `plt.plot()`
- 评估：每个 API 调用是否正确、参数是否合法

**评测维度**：
1. **API 调用正确性**
   - 参数类型是否匹配
   - 返回值是否正确处理

2. **错误处理**
   - 是否捕获了 API 可能抛出的异常
   - 异常处理是否合理

**指标计算**：

```
API 调用正确率 = (正确的 API 调用数 / 总 API 调用数) × 100%

其中：
- 正确的 API 调用：参数类型正确 + 返回值正确处理 + 无运行时错误
- 通过单元测试验证

异常处理覆盖率 = (有异常处理的 API 调用数 / 可能抛出异常的 API 调用数) × 100%

其中：
- 可能抛出异常的 API：根据 API 文档标注
- 异常处理：try-except 块或错误码检查
```

## 阶段 3：测试

### 数据集组合
- **TestGenEval**（单元测试生成）
- **EvoCodeBench**（测试覆盖率）

### 3.1 TestGenEval：单元测试生成质量

**典型示例**：
- 源代码：Python 函数 `calculate_discount(price, coupon_code)`
- 输入：函数签名 + 文档字符串
- 预期输出：完整的 pytest 测试用例（5-10 个测试函数）
- 数据规模：68,647 个测试用例，来自 11 个 Python 仓库

**评测维度**：
1. **测试用例质量**
   - 是否覆盖了正常路径和异常路径
   - 是否包含边界条件测试

2. **测试可执行性**
   - 生成的测试是否能通过语法检查
   - 是否能成功运行

**指标计算**：

```
测试通过率 = (通过的测试用例数 / 生成的测试用例总数) × 100%

其中：
- 通过的测试：语法正确 + 运行无错误 + 断言有效
- 通过 pytest 执行验证

分支覆盖率 = (覆盖的分支数 / 总分支数) × 100%

其中：
- 分支：if-else, switch-case, 循环等控制流分支
- 使用 coverage.py 工具测量

变异测试得分 = (杀死的变异体数 / 总变异体数) × 100%

其中：
- 变异体：对源代码进行微小修改（如 + 改为 -）后的版本
- 杀死：测试用例能检测到该变异（测试失败）
- 使用 mutmut 工具生成变异体
```

### 3.2 EvoCodeBench：测试覆盖率评估

**典型示例**（复用阶段 2 的 Database 领域 B+ 树实现）：
- 源代码：B+ 树的插入、查询、删除函数
- 生成测试：针对这些函数的测试用例
- 评估：测试是否覆盖了所有边界条件（空树、单节点、满节点、分裂、合并等）

**指标计算**：

```
语句覆盖率 = (执行的语句数 / 总语句数) × 100%

路径覆盖率 = (执行的路径数 / 总路径数) × 100%

其中：
- 路径：从函数入口到出口的所有可能执行路径
- 总路径数：通过控制流图计算（可能指数级增长）
- 实际中通常采样关键路径
```

## 阶段 4：维护

### 数据集组合
- **RepoDebug**（仓库级调试）
- **AACR-Bench**（代码审查）

### 4.1 RepoDebug：Bug 定位与修复

**典型示例**：
- 仓库：`requests` 库（Python HTTP 客户端）
- Bug 类型：`NullPointerException` 在 `Session.send()` 方法
- 错误信息：堆栈跟踪 + 失败的测试用例
- 涉及文件：3-5 个相关文件（跨文件依赖）
- 数据规模：8 种编程语言，22 种错误类型

**评测维度**：
1. **Bug 定位准确性**
   - 是否定位到正确的文件和行号
   - 是否识别了根本原因（而非表面症状）

2. **修复策略合理性**
   - 修复是否解决了根本问题
   - 是否引入了新的 Bug

3. **跨文件追踪能力**
   - 是否追踪了跨文件的调用链
   - 是否理解了模块间的依赖关系

**指标计算**：

```
Bug 定位准确率 = (正确定位的 Bug 数 / 总 Bug 数) × 100%

其中：
- 正确定位：定位到的文件和行号在专家标注的范围内（±5 行）

修复成功率 = (成功修复的 Bug 数 / 尝试修复的 Bug 数) × 100%

其中：
- 成功修复：修复后所有测试通过 + 未引入新 Bug（通过回归测试验证）

跨文件追踪深度 = Σ(Bug i 涉及的文件数) / 总 Bug 数

其中：
- 涉及的文件数：从 Bug 触发点到根本原因的调用链中的文件数
- 平均值越高，说明模型的跨文件理解能力越强
```

### 4.2 AACR-Bench：代码审查质量（含人工标注）

**典型示例**：
- Pull Request：为 `django` 项目添加新的 ORM 查询优化
- 代码变更：修改了 3 个文件，新增 200 行代码
- 审查任务：发现潜在缺陷、性能问题、安全漏洞
- 数据规模：AI 辅助 + 专家验证，发现缺陷数比传统方法多 285%

**评测维度**：
1. **缺陷发现能力**
   - 是否发现了所有专家标注的缺陷
   - 是否发现了专家未标注的新缺陷

2. **审查质量**（需人工评估）
   - 审查意见是否具体、可操作
   - 是否提供了修复建议
   - 意见分类是否准确（Bug/Style/Design/Performance）

3. **跨文件上下文理解**
   - 是否理解了代码变更对其他模块的影响
   - 是否发现了跨文件的一致性问题

**指标计算**：

```
缺陷召回率 = (发现的真实缺陷数 / 专家标注的缺陷总数) × 100%

缺陷精确率 = (发现的真实缺陷数 / 报告的缺陷总数) × 100%

其中：
- 真实缺陷：专家确认的缺陷
- 报告的缺陷：模型报告的所有缺陷（包括误报）

F1 得分 = 2 × (精确率 × 召回率) / (精确率 + 召回率)

审查意见质量 = Σ(意见 i 的可操作性得分) / 审查意见总数

其中：
- 可操作性得分：专家评分（1-5 分）
  - 5 分：具体指出问题 + 提供修复建议 + 解释原因
  - 3 分：指出问题但缺少修复建议
  - 1 分：模糊的意见，无法操作

意见分类准确率 = (分类正确的意见数 / 总意见数) × 100%

其中：
- 分类：Bug, Style, Design, Performance, Security
- 需专家标注每条意见的正确分类
```

**实现路径**：
- 半自动：AACR-Bench 提供基础数据
- 人工标注：3-5 名开发者标注审查意见质量（约 2-4 周）
- 需定义标注规范和质量评分标准

## 各阶段综合得分计算

**阶段 1（需求与设计）**：
```
需求设计得分 = 0.25 × API 选择合理性 + 0.35 × 模块内聚度 + 0.25 × 领域需求覆盖率 + 0.15 × 歧义识别率
```

**阶段 2（构造）**：
```
构造得分 = 0.3 × (1 - 实现异味密度/10) + 0.3 × 时间效率 + 0.2 × 内存效率 + 0.2 × API 调用正确率
```

**阶段 3（测试）**：
```
测试得分 = 0.4 × 变异测试得分 + 0.3 × 分支覆盖率 + 0.3 × 测试通过率
```

**阶段 4（维护）**：
```
维护得分 = 0.3 × Bug 定位准确率 + 0.3 × 修复成功率 + 0.2 × 缺陷 F1 得分 + 0.2 × 审查意见质量
```

## v0.4/v0.5 中提到的新指标补充

### 5.1 部分正确性（Partial Correctness）

**定义**：当代码无法完全通过所有测试用例时，衡量其通过的测试用例比例。

**适用场景**：LiveCodeBench、BigCodeBench等包含多个测试用例的数据集

**指标计算**：

```
部分正确性 = (通过的测试用例数 / 总测试用例数) × 100%

其中：
- 通过的测试用例：执行成功且输出与预期一致的测试用例
- 总测试用例：数据集为该任务提供的所有测试用例（包括公开和隐藏）
- 典型值范围：40-90%
```

**与pass@k的区别**：
- pass@k：二值判定（全部通过/不通过）
- 部分正确性：连续值，能区分"差一点通过"和"完全错误"

---

### 5.2 执行效率归一化（Normalized Execution Time, NET）

**定义**：生成代码的执行时间相对于最优解（或基准解）的比值。

**适用场景**：LiveCodeBench、EvoCodeBench等包含性能测试的数据集

**指标计算**：

```
NET = 生成代码执行时间 / 基准解执行时间

执行效率得分 = 1 / NET

其中：
- 基准解：数据集提供的最优解或专家编写的参考解
- NET = 1：与基准解性能相同
- NET > 1：性能劣于基准解
- NET < 1：性能优于基准解（罕见）
- 典型值范围：NET ∈ [0.8, 3.0]
```

**实现路径**：
- 全自动：在沙箱环境中执行代码，记录运行时间
- 工具：time命令、cProfile（Python）、perf（Linux）
- 注意事项：多次运行取中位数，避免系统抖动影响

---

### 5.3 补丁最小性（Patch Minimality）

**定义**：生成的Bug修复补丁相对于金标准补丁的行数比值。

**适用场景**：RepoDebug等Bug修复数据集

**指标计算**：

```
补丁最小性比值 = 生成补丁行数 / 金标准补丁行数

补丁最小性得分 = 1 / max(补丁最小性比值, 1)

其中：
- 生成补丁行数：模型生成的修复代码的行数（diff格式）
- 金标准补丁：数据集提供的专家修复方案
- 比值 = 1：与金标准相同
- 比值 > 1：修改过多（可能引入不必要的变更）
- 比值 < 1：修改过少（可能未完全修复）
- 典型值范围：0.8-2.5
```

**评估意义**：
- 补丁越小，引入新Bug的风险越低
- 补丁越小，代码审查成本越低
- 但过小的补丁可能未完全修复问题

---

### 5.4 跨文件一致性检测（Cross-File Consistency Detection）

**定义**：在代码审查中，模型能否发现跨文件的一致性问题（如接口变更未同步更新调用方）。

**适用场景**：AACR-Bench等代码审查数据集

**指标计算**：

```
跨文件一致性检测率 = (发现的跨文件问题数 / 标注的跨文件问题总数) × 100%

其中：
- 跨文件问题：涉及多个文件的一致性问题
  - 接口签名变更但调用方未更新
  - 数据结构定义变更但使用方未同步
  - 配置文件与代码逻辑不一致
- 标注的跨文件问题：专家标注的所有跨文件一致性问题
- 典型值范围：30-70%
```

**实现路径**：
- 半自动：AACR-Bench提供基础数据
- 人工标注：需专家标注哪些问题属于跨文件一致性问题
- 人力需求：2-3人，1-2周

---

### 5.5 根因分析准确性（Root Cause Analysis Accuracy）

**定义**：在Bug定位中，模型能否识别Bug的根本原因（而非表面症状）。

**适用场景**：RepoDebug等Bug修复数据集

**指标计算**：

```
根因分析准确性 = (根因分析正确的Bug数 / 总Bug数) × 100%

其中：
- 根因分析正确：模型输出的根因描述与专家标注一致
- 评估方法：
  - 自动：关键词匹配（低精度）
  - 半自动：LLM-as-Judge对比模型输出与专家标注（中精度）
  - 人工：专家评估（高精度）
- 典型值范围：40-75%
```

**评分标准**（人工评估）：
- 5分：准确识别根本原因，解释清晰
- 3分：识别了部分原因，但不够深入
- 1分：仅描述表面症状，未触及根因

---

### 5.6 指令遵循率（Instruction Following Rate, IFR）

**定义**：生成代码遵循任务描述中约束条件的比例（v1.0中已提及，此处补充详细计算）。

**适用场景**：BigCodeBench等包含约束条件的数据集

**指标计算**（详细版）：

```
IFR = (IFR_hard × w_hard + IFR_runtime × w_runtime + IFR_soft × w_soft) / (w_hard + w_runtime + w_soft)

其中：
IFR_hard    = 通过的硬约束数 / 硬约束总数
IFR_runtime = 通过的运行时约束数 / 运行时约束总数
IFR_soft    = 通过的软约束数 / 软约束总数

权重建议：
w_hard = 3（硬约束最重要）
w_runtime = 2（运行时约束次之）
w_soft = 1（软约束权重最低）

约束类型示例：
- 硬约束：函数签名必须是 def foo(x: int) -> str
- 运行时约束：执行时间不超过1秒
- 软约束：使用驼峰命名法
```

**验证方法**：
- 硬约束：tree-sitter AST检查（全自动）
- 运行时约束：执行验证（全自动）
- 软约束：LLM-as-Judge（半自动）

## 场景化权重配置（v1.1新增）

不同评测场景下，各阶段的权重配置不同：

| 评测场景 | 需求与设计 | 软件构造 | 测试 | 维护 | 说明 |
|---------|-----------|---------|------|------|------|
| **通用评测** | 20% | 40% | 20% | 20% | 平衡各阶段能力 |
| **企业选型** | 15% | 30% | 15% | 40% | 重视维护能力（Bug修复、代码审查） |
| **学术研究** | 30% | 35% | 20% | 15% | 重视需求理解和算法设计 |
| **算法竞赛** | 5% | 70% | 15% | 10% | 重视功能正确性和执行效率 |
| **Agent专项** | 25% | 25% | 20% | 30% | 重视多轮交互和跨文件理解 |

**使用方法**：
```
总分 = Σ (阶段权重 × 阶段内结果指标平均分)

其中：
- 阶段权重：根据评测场景选择（见上表）
- 阶段内结果指标：仅包含Outcome层指标，按权重加权平均
- 诊断指标：不参与计分，仅在报告中展示
- 成本指标：单独报告，不计入总分
```

## 人工标注数据构建方案

### 6.1 API选择合理性标注

**目标**：为BigCodeBench中的任务标注最优API选择方案。

**标注流程**：
1. **种子数据采集**：从BigCodeBench抽样200个任务
2. **专家标注**：3名资深开发者独立标注每个任务的最优API选择
3. **一致性检验**：计算Fleiss' Kappa，要求≥0.7
4. **冲突解决**：对不一致的标注进行讨论，达成共识
5. **扩展标注**：基于种子数据训练LLM-as-Judge，扩展到全量数据

**标注规范**：
- 最优API：功能最匹配、性能最优、代码最简洁的API
- 冗余依赖：引入了但未使用的库
- 标注格式：JSON，包含任务ID、最优API列表、冗余依赖列表

**人力需求**：3人×2周 = 6人周

---

### 6.2 领域需求覆盖标注

**目标**：为EvoCodeBench的10个领域定义核心功能清单。

**标注流程**：
1. **领域专家招募**：每个领域招募1-2名专家
2. **功能清单定义**：专家定义该领域的核心功能（5-10个）
3. **示例标注**：为每个功能提供正例和反例
4. **交叉验证**：不同专家交叉验证功能清单的完整性

**标注示例**（Database领域）：
```json
{
  "domain": "Database",
  "core_functions": [
    "CRUD操作（Create, Read, Update, Delete）",
    "事务管理（Transaction, Commit, Rollback）",
    "索引优化（Index, B+Tree）",
    "查询优化（Query Optimization）",
    "并发控制（Lock, MVCC）"
  ]
}
```

**人力需求**：10人×1周 = 10人周

---

### 6.3 Bug定位专家标注

**目标**：为RepoDebug中的Bug标注正确的定位范围（文件、函数、行号）。

**标注流程**：
1. **数据抽样**：从RepoDebug抽样100个Bug
2. **专家定位**：2名开发者独立定位每个Bug
3. **一致性检验**：计算Cohen's Kappa，要求≥0.8
4. **标注格式**：JSON，包含Bug ID、文件路径、函数名、行号范围（±5行）

**标注规范**：
- 文件级：Bug所在的文件
- 函数级：Bug所在的函数
- 行级：Bug的具体行号（允许±5行误差）

**人力需求**：2人×2周 = 4人周

---

### 6.4 代码审查意见质量标注

**目标**：为AACR-Bench中的审查意见标注质量评分和分类。

**标注流程**：
1. **数据抽样**：从AACR-Bench抽样150条审查意见
2. **质量评分**：3名开发者独立评分（1-5分）
3. **意见分类**：标注意见类型（Bug/Style/Design/Performance/Security）
4. **一致性检验**：计算Fleiss' Kappa，要求≥0.7
5. **扩展标注**：基于种子数据训练LLM-as-Judge

**评分标准**：
- 5分：具体指出问题 + 提供修复建议 + 解释原因
- 4分：具体指出问题 + 提供修复建议
- 3分：指出问题但缺少修复建议
- 2分：问题描述模糊，难以理解
- 1分：无关或错误的意见

**人力需求**：3人×2周 = 6人周
