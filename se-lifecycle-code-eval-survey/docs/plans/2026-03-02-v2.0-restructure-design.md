# 软件生命周期视角下的LLM细粒度评测提案 v2.0 - 重构设计文档

**设计日期**：2026-03-02
**版本**：v2.0 Design
**设计目标**：将v1.1文档从"生命周期→数据集和评估维度→具体指标"重构为"软件生命周期维度→评估的能力维度→具体指标→可用数据和状态"

---

## 一、设计理念

### 1.1 理论基础

本次重构基于**SWEBOK（Software Engineering Body of Knowledge）**软件工程知识体系，将LLM代码生成能力映射到软件工程的核心知识领域：

- **Software Requirements**：需求理解与规约
- **Software Design**：架构设计与评估
- **Software Construction**：功能实现与质量保证
- **Software Testing**：测试生成与验证
- **Software Maintenance**：程序理解与维护活动

### 1.2 核心变化

**v1.1结构**：
```
生命周期阶段 → 数据集 → 评估维度 → 具体指标
```

**v2.0结构**：
```
生命周期阶段 → 能力维度（基于SWEBOK） → 具体指标 → 数据集 + 状态
```

**关键改进**：
1. **能力导向**：从数据集驱动转向能力驱动，更符合评测本质
2. **理论支撑**：基于SWEBOK提供科学依据，增强学术严谨性
3. **状态透明**：明确标注每个指标的数据集可用性和自动化程度
4. **可扩展性**：能力维度可独立扩展，不受数据集限制

## 二、能力维度划分

### 2.1 阶段1：需求与设计

**理论依据**：SWEBOK Requirements + Design KAs

| 能力维度 | 子能力 | SWEBOK映射 |
|---------|--------|-----------|
| **需求理解能力** | 需求获取、需求分析、领域理解 | Requirements Elicitation & Analysis |
| **需求规约能力** | 需求明确化、歧义识别与澄清 | Requirements Specification & Validation |
| **架构设计能力** | 系统架构、模块划分、技术选型 | Software Architecture & Detailed Design |
| **设计评估能力** | 设计质量分析、设计异味检测 | Design Quality Analysis & Evaluation |

### 2.2 阶段2：构造

**理论依据**：SWEBOK Construction + Quality KAs

| 能力维度 | 子能力 | SWEBOK映射 |
|---------|--------|-----------|
| **功能实现能力** | 功能正确性、API调用正确性 | Construction Fundamentals |
| **质量保证能力** | 代码质量、执行效率、错误处理 | Construction Quality + Software Quality |

**质量保证能力细分**：
- 代码质量：可读性、可维护性、规范性
- 执行效率：时间效率、空间效率
- 错误处理：异常处理、边界条件

### 2.3 阶段3：测试

**理论依据**：SWEBOK Testing KA

| 能力维度 | 子能力 | SWEBOK映射 |
|---------|--------|-----------|
| **测试生成能力** | 测试用例生成、测试可执行性 | Test Techniques |
| **测试覆盖能力** | 语句覆盖、分支覆盖、路径覆盖 | Test Levels |
| **测试有效性能力** | 变异测试、缺陷检测能力 | Test-Related Measures |

### 2.4 阶段4：维护

**理论依据**：SWEBOK Maintenance KA

| 能力维度 | 子能力 | SWEBOK映射 |
|---------|--------|-----------|
| **程序理解能力** | 代码理解、跨文件追踪、依赖分析 | Program Comprehension |
| **纠正性维护能力** | Bug定位、根因分析、Bug修复 | Corrective Maintenance |
| **完善性维护能力** | 代码审查、缺陷发现、质量改进建议 | Perfective Maintenance |

```markdown
- 阶段3：测试

[扁平化表格，支持点击跳转]

[层级化结构，每个指标详细展开]
```

### 3.2 速查表设计（框架导向）

**设计理念**：速查表聚焦评测框架本身，数据集作为实现参考

**表格列**：
- 生命周期阶段
- 能力维度（可点击跳转）
- 评估目标（该能力要评估什么）
- 评估方法（如何评估）
- 指标类别（需要哪类指标）
- 可用数据集及状态（实现参考，可扩展）

**示例**：
```markdown
| 生命周期阶段 | 能力维度 | 评估目标 | 评估方法 | 指标类别 | 可用数据集及状态 |
|------------|---------|---------|---------|---------|-----------------|
| 需求与设计 | [需求理解能力](#11-需求理解能力) | 准确理解领域需求 | 功能覆盖度分析 | 覆盖率类 | EvoCodeBench (✅) |
| 需求与设计 | [需求理解能力](#11-需求理解能力) | 识别需求歧义 | 歧义检测与澄清 | 识别率类 | 需构建 (🔴) |
```

**状态标识**：
- ✅ 可用：数据集已发布可直接使用
- 🔴 需构建：需要自建数据集
- 🟡 部分可用：数据集存在但需额外处理

### 3.3 详细说明设计（框架导向）

**层级结构**：
```markdown
## 阶段1：需求与设计

### 1.1 需求理解能力
**理论依据**：SWEBOK Requirements KA - Requirements Elicitation & Analysis

**评估目标**：评估模型能否准确理解和把握软件需求，包括领域知识理解、需求完整性识别、需求歧义处理等方面。

**评估维度**：
1. **领域知识理解**：模型对特定领域（如数据库、网络、算法等）的理解深度
2. **需求完整性识别**：能否识别需求描述中的缺失或不完整部分
3. **需求歧义处理**：能否识别并澄清需求中的歧义表述

**评估方法**：
- 功能覆盖度分析：检查生成代码是否实现了领域核心功能
- 需求澄清能力测试：给定歧义需求，观察模型是否主动提问澄清

**指标类别**：
- **覆盖率类指标**：领域需求覆盖率、功能完整度
- **识别率类指标**：歧义识别率、缺失识别率

**实现参考**：
- **可用数据集**：
  - EvoCodeBench (✅ 可用)：包含10个领域的编程任务，可用于评估领域知识理解
  - 状态：已发布，可直接使用
  - 局限性：缺少需求歧义场景
- **需构建数据**：
  - 需求歧义测试集 (🔴 需构建)：包含歧义、不完整、矛盾需求的测试集
  - 构建方法：从Stack Overflow采集"需要更多信息"类问题
  - 人力需求：2-3人，2-4周
- **自动化程度**：🟡 半自动（需人工标注领域核心功能清单）

**可扩展性说明**：
- 数据集可替换：任何包含领域任务的数据集都可用于评估
- 指标可扩展：可增加需求理解深度、需求变更适应性等指标
- 评估方法可优化：可引入LLM-as-Judge进行自动化评估

- [返回速查表](#能力维度速查表)
```

### 3.4 导航设计

**三种跳转链接**：

## 七、可扩展性设计

### 7.1 能力维度扩展

**扩展原则**：
- 新增能力维度应有SWEBOK或其他软件工程理论支撑
- 能力维度应与生命周期阶段对应
- 避免能力维度之间的重叠

**扩展示例**：
- 构造阶段可增加"安全编码能力"（Security Coding）
- 维护阶段可增加"适应性维护能力"（Adaptive Maintenance）

### 7.2 指标扩展

**扩展原则**：
- 新增指标应归属到明确的能力维度
- 指标应有清晰的定义和计算公式
- 标注数据集状态和自动化程度

**扩展流程**：
1. 确定指标所属的能力维度
2. 在速查表中添加新行
3. 在详细说明中添加指标详情
4. 建立导航链接

### 7.3 数据集扩展

**扩展原则**：
- 新增数据集应在"数据集详情"章节中详细说明
- 在速查表中标注数据集状态
- 在相关指标中添加数据集链接

## 八、设计决策记录

### 8.1 为什么选择混合结构？

**决策**：采用速查表 + 详细说明的混合结构

**理由**：
- 速查表满足快速查找需求
- 详细说明保留学术严谨性
- 两者通过链接无缝衔接

**替代方案**：
- 纯扁平化表格：失去详细说明
- 纯层级结构：不便于快速查找

### 8.2 为什么基于SWEBOK？

**决策**：使用SWEBOK作为能力维度划分的理论基础

**理由**：
- SWEBOK是国际公认的软件工程知识体系
- 提供科学依据，增强学术严谨性
- 与软件生命周期天然对应

**替代方案**：
- ISO/IEC 25010：更侧重质量属性，不够全面
- CMMI：更侧重过程成熟度，不适合能力评测

### 8.3 为什么标注状态和自动化程度？

**决策**：在速查表中明确标注数据集状态和自动化程度

**理由**：
- 透明化：让用户清楚知道哪些指标可以立即使用
- 实用性：帮助用户评估实施成本
- 可扩展性：为未来数据集建设提供指引
