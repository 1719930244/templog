# AI 代码生成细粒度评测方案（数据集驱动视角）

> **版本**: v0.5 | **日期**: 2026-03-02

## 核心思路

以数据集为入口 → 明确可评测指标 → 给出评测流程 → 展示综合打分机制

## 二、数据集驱动的评测流程

### 2.1 LiveCodeBench v6

**数据类型**：
- 1,000+ 道竞赛算法题（2023-2025持续更新）
- 每题包含：问题描述、标准输入输出、隐藏测试用例
- 难度分层：Easy / Medium / Hard

**人工操作**：
1. 用模型生成n个代码样本（建议n=10）
2. 在沙箱环境中执行代码，收集测试通过情况
3. 对比生成代码与最优解的运行时间/内存
4. 运行Lint工具和安全扫描工具

**指标打分**：
- **pass@k**：`1 - C(n-c, k) / C(n, k)`（n个样本中c个通过，计算至少k个通过的概率）
- **部分正确性**：`通过的测试用例数 / 总测试用例数`
- **执行效率NET**：`生成代码运行时间 / 最优解运行时间`（归一化：`1 / NET`）
- **Lint密度**：`违规数 / KLOC`（归一化：`1 - min(density/10, 1)`）

---

### 2.2 BigCodeBench

**数据类型**：
- 1,140个真实世界任务（需调用139个Python库）
- 每题包含：任务描述、约束条件（硬约束/软约束）、测试用例
- 任务类型：数据处理、API调用、文件操作等

**人工操作**：
1. 生成代码并执行功能测试
2. 用AST解析检查硬约束（如禁用某些API）
3. 用LLM-as-Judge评估软约束（如代码风格）
4. 检查import语句和API调用正确性

**指标打分**：
- **pass@k**：同LiveCodeBench
- **指令遵循率IFR**：`(硬约束通过数 + 软约束通过数) / 总约束数`
- **API使用正确性**：`正确调用的API数 / 总API调用数`

---

### 2.3 RepoDebug

**数据类型**：
- 多仓库级调试任务（8种语言：Python/Java/C++/Go/JS/TS/Rust/C#）
- 22种错误类型：逻辑错误、类型错误、API误用、并发问题等
- 每个任务包含：错误描述、仓库上下文、预期修复

**人工操作**：
1. 让模型定位bug（输出文件名、函数名、行号）
2. 让模型分析根因（输出错误原因描述）
3. 让模型生成修复补丁
4. 执行测试验证修复正确性
5. 检查是否引入新问题（回归测试）

**指标打分**：
- **Bug定位准确率**：
  - 文件级：`定位正确的文件数 / 总任务数`
  - 函数级：`定位正确的函数数 / 总任务数`
  - 行级：`定位正确的行数 / 总任务数`
- **根因分析准确性**：人工评估或LLM-as-Judge（与标准答案对比）
- **修复正确性**：`修复后测试通过的任务数 / 总任务数`
- **补丁最小性**：`生成补丁行数 / 金标准补丁行数`（归一化：`1 / ratio`）

---

### 2.4 AACR-Bench

**数据类型**：
- 真实PR的代码审查任务（多语言、跨文件）
- 每个任务包含：PR diff、仓库上下文、专家标注的缺陷列表
- 缺陷类型：显性缺陷（原PR中标注）+ 潜在缺陷（专家发现的隐藏问题）

**人工操作**：
1. 让模型审查PR，输出发现的问题列表
2. 对比模型输出与专家标注
3. 统计True Positive、False Positive、False Negative
4. 评估问题描述质量和修复建议可行性

**指标打分**：
- **缺陷发现率**：
  - 显性缺陷：`TP_explicit / (TP_explicit + FN_explicit)`
  - 潜在缺陷：`TP_latent / (TP_latent + FN_latent)`
- **误报率**：`FP / (TP + FP)`
- **审查质量**：人工评估或LLM-as-Judge（问题描述准确性、修复建议可行性）
- **跨文件一致性检测**：`发现的跨文件问题数 / 标注的跨文件问题数`

## 三、细粒度评测的核心：指标分层与去冗余

### 3.1 为什么需要分层？

**问题**：45个指标全部平等计分 → 重复计分 + 掩盖真实问题

**解决方案**：分为3层

```
┌─────────────────────────────────────┐
│  诊断指标 (Diagnostic)               │
│  不参与综合评分，仅用于问题归因      │
│  ├─ Lint违规数                       │
│  ├─ 圈复杂度                         │
│  ├─ 代码异味                         │
│  └─ 代码重复率                       │
└─────────────────────────────────────┘
           ↓ 聚合影响
┌─────────────────────────────────────┐
│  结果指标 (Outcome)                  │
│  参与综合评分，衡量最终交付质量      │
│  ├─ pass@k (功能正确性)              │
│  ├─ 缺陷密度 (工程质量)              │
│  ├─ Bug定位准确率 (维护能力)         │
│  └─ 多轮交互效率 (Agent能力)         │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│  成本/体验指标 (Cost & Experience)   │
│  独立维度报告，不与质量指标混合      │
│  ├─ 生成成本 (token/费用)            │
│  └─ 开发者体验 (采纳率/时间节省)     │
└─────────────────────────────────────┘
```

### 3.2 去冗余规则

1. **诊断指标不进入综合评分**：Lint、复杂度仅用于解释"为什么缺陷密度高"
2. **同源指标取一**：Lint与代码异味相关性r>0.7，只取一个
3. **成本独立报告**：不与功能/质量线性加权

## 四、综合打分机制（详细示例）

### 4.1 评分公式

```
总分 = Σ (阶段权重 × 阶段内结果指标平均分)

其中：
- 阶段权重：根据评测场景配置（见4.2）
- 阶段内结果指标：仅包含Outcome层指标，等权平均
- 诊断指标：不参与计分，仅在报告中展示
- 成本指标：单独报告，不计入总分
```

### 4.2 场景化权重配置

| 评测场景 | 需求 | 实现 | 测试 | 维护 | Agent | 通用 |
|---------|------|------|------|------|-------|------|
| **通用评测** | 5% | 40% | 20% | 20% | 10% | 5% |
| **企业选型** | 10% | 30% | 15% | 30% | 10% | 5% |
| **学术研究** | 15% | 35% | 20% | 15% | 10% | 5% |
| **Agent专项** | 5% | 20% | 10% | 15% | 45% | 5% |

### 4.3 完整评测示例：GPT-4 on LiveCodeBench v6

#### Step 1: 原始指标采集

```json
{
  "dataset": "LiveCodeBench v6",
  "model": "GPT-4",
  "sample_size": 1000,
  "raw_metrics": {
    "M-2.1.1_pass@1": 0.72,
    "M-2.1.1_pass@5": 0.85,
    "M-2.1.2_partial": 0.81,
    "M-2.2.1_lint_density": 2.3,
    "M-2.2.2_complexity": 8.2,
    "M-2.2.4_smell_density": 1.8,
    "M-2.3.1_NET": 1.15,
    "M-2.4.1_vuln_rate": 0.02,
    "M-6.3_cost_per_task": 0.05
  }
}
```

#### Step 2: 指标归一化（0-1区间）

```python
# 归一化规则
normalized = {
    "pass@1": 0.72,  # ↑ 越高越好，直接使用
    "partial": 0.81,  # ↑ 越高越好，直接使用
    "lint_density": 1 - min(2.3/10, 1),  # ↓ 越低越好，反向归一化
    "complexity": 1 - min(8.2/20, 1),  # ↓ 越低越好，反向归一化
    "NET": 1 / 1.15,  # ↓ 越低越好，倒数归一化
    "vuln_rate": 1 - 0.02  # ↓ 越低越好，反向归一化
}
```

#### Step 3: 指标分层

```python
# 结果指标（参与综合评分）
outcome_metrics = {
    "pass@1": 0.72,
    "partial": 0.81,
    "NET_normalized": 0.87,
    "vuln_rate_normalized": 0.98
}

# 诊断指标（不参与评分，仅展示）
diagnostic_metrics = {
    "lint_density": 2.3,
    "complexity": 8.2,
    "smell_density": 1.8
}

# 成本指标（独立报告）
cost_metrics = {
    "cost_per_task": 0.05,
    "total_cost": 50.0
}
```

#### Step 4: 阶段内平均

```python
# 实现阶段结果指标平均
implementation_score = (0.72 + 0.81 + 0.87 + 0.98) / 4 = 0.845
```

#### Step 5: 跨数据集指标收集（完整评测）

从不同数据集收集各阶段的结果指标，计算阶段平均分：

- **需求阶段**（BigCodeBench）：IFR_hard + IFR_soft → 平均
- **实现阶段**（LiveCodeBench）：pass@1 + partial + NET + vuln_rate → 平均
- **测试阶段**（TestGenEval）：test_quality + coverage_gain + mutation_score → 平均
- **维护阶段**（RepoDebug + AACR-Bench）：bug_location + fix_correctness + defect_detection → 平均
- **Agent能力**（跨数据集）：success_at_N + cross_file_consistency → 平均
- **通用质量**（跨数据集）：defect_density + maintainability_index → 平均

#### Step 6: 综合打分（通用评测场景）

使用场景化权重配置（见下表），计算加权总分：

```
总分 = Σ (阶段权重 × 阶段平均分)
```

**权重配置示例**（通用评测）：
- 需求5% + 实现40% + 测试20% + 维护20% + Agent10% + 通用5% = 100%

**场景化权重配置**：

| 评测场景 | 需求 | 实现 | 测试 | 维护 | Agent | 通用 |
|---------|------|------|------|------|-------|------|
| **通用评测** | 5% | 40% | 20% | 20% | 10% | 5% |
| **企业选型** | 10% | 30% | 15% | 30% | 10% | 5% |
| **学术研究** | 15% | 35% | 20% | 15% | 10% | 5% |
| **Agent专项** | 5% | 20% | 10% | 15% | 45% | 5% |

## 六、常见问题

**Q1: 为什么选择LiveCodeBench？**
A: 持续更新（2023-2025），避免数据污染，题目来自真实竞赛。

**Q2: 如何对比不同模型？**
A: 使用相同数据集+相同权重配置，对比综合得分和各阶段得分。

**Q3: 诊断指标有什么用？**
A: 当综合得分低时，诊断指标帮助定位问题根因（如"pass@k低是因为复杂度过高"）。

**Q4: RepoDebug和AACR-Bench如何互补？**
A: RepoDebug评测"修复已知bug"的能力，AACR-Bench评测"发现潜在问题"的能力，两者覆盖维护阶段的不同方面。
