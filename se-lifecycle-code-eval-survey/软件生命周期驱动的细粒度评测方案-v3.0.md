## 0. 框架总览

### 0.1 设计原则

本框架遵循四项核心设计原则：

1. **科学性**：能力分解以 SWEBOK（Software Engineering Body of Knowledge）知识域为理论依据，12 项能力覆盖 Requirements KA、Construction KA、Testing KA、Maintenance KA 四大知识域，确保评测维度具有工程理论基础而非经验拼凑
2. **可扩展性**：框架提供能力、指标、数据集、阶段四类正式扩展点，新增指标通过注册表接入而非修改框架结构，支持领域定制和版本演进
3. **企业级**：评测结果可作为模型选型的量化依据，框架支持跨企业对标和数据资产沉淀
4. **数据集质量感知**：所有推荐数据集均经过六维质量评估（学术严谨性、工业采纳度、维护状态、数据质量、规模与多样性、工具支持），标注质量评级和风险提示

### 0.2 四层层次结构

```
Level 1: 生命周期阶段 (Stage)          — 4 个
  Level 2: 能力 (Capability)           — 每阶段 3 个，共 12 个
    Level 3: 指标 (Metric)             — 每能力 2-4 个，共 35 个
      Level 4: 数据集/基准 + 基建       — 每指标关联数据集 + 工具链
```

**命名规范**：
- 能力 ID：`C-{阶段}.{序号}`，如 C-1.1 表示阶段 1 第 1 项能力
- 指标 ID：`M-{阶段}.{能力}.{序号}`，如 M-1.1.1 表示 C-1.1 能力下第 1 个指标
- 指标方向：↑ 越高越好 / ↓ 越低越好
- 自动化级别：🟢 全自动 / 🟡 半自动 / 🔴 人工

#### 阶段 1：需求与设计

> SWEBOK 映射：Requirements Elicitation & Analysis, Requirements Specification, Software Architecture & Detailed Design

##### C-1.1 需求理解能力

**评估目标**：模型对领域知识的理解深度、需求完整性的把握、歧义和矛盾的识别能力。需求理解是软件开发的起点，直接决定了后续生成代码的方向正确性。当前大语言模型在通用编程任务上的表现已趋于成熟，但在垂直领域（如数据库、安全、嵌入式等）中是否具备足够的领域知识储备，以及在面对模糊或相互矛盾的需求描述时能否主动识别问题而非盲目生成代码，仍是区分模型能力层次的关键维度。

**M-1.1.1 领域需求覆盖率** ↑

*coverage = |实现的领域核心功能| / |领域核心功能总数| × 100%*

每个领域（如 Database、Security、GUI 等）预定义 5-10 个核心功能清单，检查生成代码是否实现了这些功能。

- **自动化级别**：🟡 半自动，功能清单需专家定义，匹配可自动化
- **工具链**：tree-sitter AST 分析 + 关键词匹配 + LLM-as-Judge
- **典型值**：50%-90%

**M-1.1.2 领域特定改进 DSI** ↑

*DSI = (模型在目标领域的 Pass@1 - 模型在所有领域的平均 Pass@1) / 模型在所有领域的平均 Pass@1*

DSI > 0 表示模型的优势领域，DSI < 0 表示模型的薄弱领域。

- **自动化级别**：🟢 全自动，基于 pass@1 计算
- **工具链**：沙箱执行环境 + 统计脚本
- **典型值**：-0.3 至 +0.3

**M-1.1.3 歧义识别率** ↑

*ambiguity_detection_rate = |识别出的歧义需求| / |总歧义需求| × 100%*

当需求存在歧义、不完整或矛盾时，模型是否能识别并主动澄清。

- **自动化级别**：🔴 人工，需构建含歧义需求的测试集，人工判定模型响应
- **工具链**：LLM-as-Judge 初筛 + 人工终审
- **典型值**：30%-70%

**推荐数据集**

- **EvoCodeBench** (A-)：275 任务覆盖 10 个编程领域，半年更新防污染，NeurIPS 2024
- **DevBench** (B)：唯一覆盖需求到设计到实现全流程的数据集，ACL 2024

**基建需求**

- 领域核心功能清单：每领域 1-2 名专家定义，约 10 人周
- 歧义需求数据集：从 Stack Overflow 采集种子数据 + LLM 扩增 + 人工审核，约 6-12 人周

##### C-1.2 指令遵循能力

**评估目标**：生成代码对需求中显式约束（硬约束、运行时约束、软约束）的满足程度，以及代码与需求的语义一致性。在实际开发场景中，用户的需求描述往往包含多层级约束：从强制性的编译和接口要求，到运行时行为和性能限制，再到编码风格等软性偏好。模型能否在生成代码时全面遵循这些约束，是衡量其工程实用性的重要标准。现有 pass@k 指标仅判定"功能是否正确"，无法揭示模型在约束遵循维度上的具体短板，因此需要引入更细粒度的指令遵循评估。

**M-1.2.1 指令遵循率 IFR** ↑

*IFR = (IFR_hard × n_hard + IFR_runtime × n_runtime + IFR_soft × n_soft) / (n_hard + n_runtime + n_soft)*

**IFR_hard    = 通过的硬约束数 / 硬约束总数**        # AST/编译/导入/禁用 API
**IFR_runtime = 通过的运行时约束数 / 运行时约束总数**  # 行为/性能/输出格式
**IFR_soft    = 通过的软约束数 / 软约束总数**          # 风格/命名偏好

当某类约束为空时（如 n_runtime = 0），该类不计入分母。建议分别报告三类 IFR 并输出失败归因。

- **自动化级别**：🟢 全自动（硬约束 + 运行时约束）/ 🟡 半自动（软约束需 LLM 判定）
- **工具链**：tree-sitter AST 检查 + 规则引擎 + 沙箱执行
- **典型值**：60%-95%

**M-1.2.2 需求-代码语义对齐度 SBC** ↑

*SBC = w₁·cosine_sim(embed(R), embed(R')) + w₂·|matched_constraints| / |total_constraints(R)| - w₃·|extra_constraints(R')| / max(|all_constraints(R')|, 1)*

通过"逆向生成"技术评估代码对需求的忠实度：给定需求 R，模型生成代码 C，再由 LLM 从 C 反向重建需求 R'，评估 R 与 R' 的一致性。权重 w₁/w₂/w₃ 通过标注小样本（≥50 对）做回归校准确定。

- **自动化级别**：🟡 半自动，LLM 逆向生成 + embedding 计算
- **工具链**：sentence-transformers, LLM API, spaCy
- **典型值**：0.5-0.9

**推荐数据集**

- **BigCodeBench** (A-)：1,140 个复杂指令任务 + 139 库约束，ICLR 2025，有排行榜

**基建需求**

- IFR 规则引擎开发：约 2 周
- SBC 权重校准标注：≥50 对需求-代码样本，约 2 人天

##### C-1.3 架构设计能力

**评估目标**：模块划分合理性、耦合内聚度、API 选择、设计异味检测。随着 LLM 代码生成从函数级向项目级演进，模型能否产出合理的模块划分和架构方案变得至关重要。低质量的架构设计会导致后续维护成本指数级增长，而当前评测体系几乎完全忽视了这一维度。本框架采用经典的 CK 度量套件（Chidamber-Kemerer metrics）结合设计异味检测，从内聚、耦合、API 合理性和设计模式四个角度系统评估模型的架构设计能力。

**M-1.3.1 模块内聚度 LCOM** ↓

*LCOM = 1 - (Σ 方法对共享属性数 / (方法数 × 属性数))*

理想值 LCOM ≤ 0.3（高内聚）。

- **自动化级别**：🟢 全自动，OO 度量工具
- **工具链**：radon (Python), JDepend (Java), Understand (多语言)
- **适用粒度**：file / repo（仓库级代码）

**M-1.3.2 模块耦合度 CBO** ↓

*CBO = 该模块引用的外部类数 + 引用该模块的外部类数*

理想值 CBO ≤ 5。

- **自动化级别**：🟢 全自动
- **工具链**：同 M-1.3.1

**M-1.3.3 API 选择合理性** ↑

*api_rationality = |正确 API 选择数| / |总 API 调用数| × 100%*

"正确 API"由专家标注的最优 API 选择集合确定。

- **自动化级别**：🟡 半自动，需专家标注最优 API 选择
- **工具链**：AST 提取 API 调用 + 专家标注对比
- **典型值**：50%-85%

**M-1.3.4 设计异味得分** ↑

*design_smell_score = 1 - |检测到的设计异味数| / |总模块数|*

设计异味包括 Cyclic Dependency、God Class、Feature Envy 等。

- **自动化级别**：🟢 全自动
- **工具链**：Designite, SonarQube
- **典型值**：0.6-0.9

**推荐数据集**

- **DevBench** (B)：含架构设计 + 模块划分评估，ACL 2024，全 SDLC 覆盖
- **ProjDevBench** (B)：项目级多文件，2,690 样本，⚠️ 未经同行评审

**基建需求**

- 架构质量评估无专用 benchmark，需依赖静态分析工具直接度量
- API 选择合理性标注：200 任务 × 3 标注员，约 6 人周

> **阶段 1 缺口**：需求工程 benchmark 仅占现有 benchmark 总量的 5%，软件设计仅占 3%。这是当前评测生态的显著短板。

#### 阶段 2：软件构造

> SWEBOK 映射：Construction Fundamentals, Construction Quality, Software Quality - Performance

软件构造是当前 LLM 代码生成评测中研究最充分的阶段，已有超过 60% 的现存 benchmark 集中于此。本框架在继承 pass@k 等经典指标的基础上，进一步将评估维度拓展至代码质量和执行效率两个在工程实践中同等重要但长期被忽视的方向，形成功能正确性、代码质量、执行效率三位一体的构造阶段评估体系。

##### C-2.1 功能正确性能力

**评估目标**：生成代码的功能正确性，包括完全正确（pass@k）、部分正确（通过的测试比例）、API 调用正确率。功能正确性是代码生成最基本的质量要求，也是目前学术界和工业界共同关注的核心指标。本框架保留 pass@k 作为基准指标的同时，引入部分正确性和 API 调用正确率作为补充，旨在克服二值判定的信息损失问题，提供更细粒度的正确性画像。

**M-2.1.1 pass@k** ↑

*pass@k = E[1 - C(n-c, k) / C(n, k)]*

生成 n 个样本，c 个通过全部测试，从中取 k 个至少有一个正确的概率。

- **自动化级别**：🟢 全自动，代码执行 + 测试用例判定
- **工具链**：Docker 沙箱执行环境
- **典型值**：40%-85% (pass@1)

**M-2.1.2 部分正确性** ↑

*partial_score = |通过的测试用例| / |总测试用例| × 100%*

与 pass@k 的区别：pass@k 为二值判定，部分正确性为连续值，能区分"差一点通过"和"完全错误"。

- **自动化级别**：🟢 全自动
- **工具链**：同 pass@k，改为逐用例统计
- **典型值**：50%-90%

**M-2.1.3 API 调用正确率** ↑

*api_correctness = |正确的 API 调用| / |总 API 调用| × 100%*

正确的 API 调用：参数类型正确 + 返回值正确处理 + 无运行时错误。

- **自动化级别**：🟢 全自动
- **工具链**：沙箱执行 + AST API 调用提取
- **典型值**：70%-98%

**推荐数据集**

- **EvalPlus** (A)：工业标准，~5000 引用，80x/35x 增强测试，NeurIPS 2023，所有大厂使用
- **LiveCodeBench** (A)：防污染设计，1,055+ 题持续更新，ICLR 2025，有排行榜
- **BigCodeBench** (A-)：复杂 API 调用，1,140 任务，ICLR 2025，有排行榜

**基建需求**

- 沙箱执行环境搭建：Docker 多语言镜像，约 1 周
- 评测执行全自动化

##### C-2.2 代码质量能力

**评估目标**：生成代码的工程质量，包括复杂度、异味、重复率、安全漏洞。已有研究表明，约 60.9% 的 AI 生成代码单元包含至少一个代码异味，且 LLM 倾向于生成过度工程化的代码结构。在企业级应用中，代码质量直接影响后续的可维护性和团队协作效率。本框架从四个互补角度构建代码质量评估体系：圈复杂度衡量逻辑复杂性，代码异味密度反映工程规范遵循程度，代码重复率评估抽象能力，漏洞引入率则关注安全性这一底线要求。

**M-2.2.1 圈复杂度 CC** ↓

*V(G) = E - N + 2P*

E 为控制流图的边数，N 为节点数，P 为连通分量数。阈值：1-10 简单，11-20 中等，>20 复杂。

- **自动化级别**：🟢 全自动
- **工具链**：radon (Python), lizard (多语言), SonarQube
- **典型值**：3-15

**M-2.2.2 代码异味密度** ↓

*smell_density = |检出的代码异味数| / (LoC / 1000)*

AI 特有异味模式：过度工程、冗余代码、不必要的抽象、硬编码调试残留。实证数据：约 60.9% 的 AI 生成代码单元包含至少一个异味。

- **自动化级别**：🟢 全自动
- **工具链**：SonarQube, Pylint/Ruff, Semgrep（AI 特有异味规则）
- **典型值**：2-8 个/KLOC

**M-2.2.3 代码重复率** ↓

*duplication_rate = |重复代码行数| / |总代码行数| × 100%*

重复代码：连续 6 行以上的相同代码块。理想值 ≤ 5%。

- **自动化级别**：🟢 全自动
- **工具链**：PMD-CPD, SonarQube
- **典型值**：0-10%

**M-2.2.4 漏洞引入率** ↓

*vuln_rate = |含漏洞的生成样本数| / |总生成样本数| × 100%*

按 CWE 分类统计（CWE-79 XSS、CWE-89 SQL 注入等）。

- **自动化级别**：🟢 全自动
- **工具链**：Bandit (Python), Semgrep (多语言), CodeQL (语义分析)
- **典型值**：5%-25%

**推荐数据集**

- **CyberSecEval** (B+)：Meta 出品，多维安全评测，漏洞引入率专用
- 静态分析基建 (—)：Pylint/Ruff/SonarQube 直接度量，不依赖专用 benchmark

**基建需求**

- 静态分析工具链配置：约 3-5 天
- AI 特有异味 Semgrep 规则编写：约 1-2 周

##### C-2.3 执行效率能力

**评估目标**：生成代码的运行时效率和内存效率。在功能正确性之外，代码的执行效率是影响系统性能和资源成本的核心因素。实证研究表明，LLM 生成的代码在运行时间上平均慢于人类最优解 2 至 3 倍，在内存消耗上差距更大。本框架通过归一化指标将生成代码的效率与基线实现进行对比，既支持横向模型间比较，也支持纵向追踪同一模型在不同版本间的效率演进趋势。

**M-2.3.1 执行效率归一化 NET** ↓

*NET = T_generated / T_baseline*

T_baseline 可选：官方最优解、人类提交 P50/P90、已知最优公开实现（需在报告中声明）。NET < 1 表示优于基线，NET > 1 表示劣于基线。测量需控制方差：排除前 3 次预热、固定随机种子、取 5 次中位数。

- **自动化级别**：🟢 全自动
- **工具链**：Docker + cgroups 资源限制, time/perf, tracemalloc
- **典型值**：0.5-3.0

**M-2.3.2 内存效率归一化 NTMU** ↓

*NTMU = M_generated / M_baseline*

基线定义同 NET。

- **自动化级别**：🟢 全自动
- **工具链**：valgrind/tracemalloc, memory_profiler
- **典型值**：0.5-5.0

**推荐数据集**

- **Mercury** (A-)：效率评测专用，1,889 任务，NeurIPS 2024，有排行榜
- **EffiBench-X** (B+)：Mercury 的多语言扩展，NeurIPS 2025

**基建需求**

- 性能采集 pipeline 搭建：约 1 周
- 计算资源：需标准化测试环境避免硬件差异

#### 阶段 3：测试与验证

> SWEBOK 映射：Test Techniques, Test Levels, Test-Related Measures

测试与验证是保障软件质量的关键环节。LLM 在测试生成任务上展现出巨大潜力，能够基于源代码自动生成单元测试，从而降低开发者的测试编写负担。然而，生成测试的质量参差不齐：部分测试存在编译错误，部分断言过于宽松无法有效检出缺陷，还有部分测试表现出不稳定性（flaky test），这些问题严重制约了 LLM 测试生成在工业实践中的可信度。本框架从测试生成质量、覆盖深度和缺陷检测有效性三个能力维度，构建了覆盖测试全生命周期的评估体系。

##### C-3.1 测试生成能力

**评估目标**：AI 生成测试代码的构建质量、通过率和断言精确度。测试生成是一个多阶段过程：首先生成的测试代码必须能够成功编译，然后在正确实现上应当通过执行，最后其断言应当精确反映预期行为而非仅检查"不抛异常"。本框架通过三个逐层递进的指标分别评估这三个阶段的质量，帮助定位模型在测试生成流水线上的具体瓶颈。

**M-3.1.1 测试构建通过率** ↑

*build_rate = |编译通过的测试数| / |总生成测试数| × 100%*

- **自动化级别**：🟢 全自动
- **工具链**：pytest/JUnit + 沙箱执行
- **典型值**：70%-95%

**M-3.1.2 测试通过率** ↑

*pass_rate = |在正确实现上通过的测试数| / |编译通过的测试数| × 100%*

"通过"指语法正确 + 运行无错误 + 断言有效。

- **自动化级别**：🟢 全自动
- **工具链**：同 M-3.1.1
- **典型值**：60%-95%

**M-3.1.3 测试预言质量** ↑

评估生成的断言是否准确反映预期行为，而非仅检查"不抛异常"。通过断言密度（每个测试方法的 assert 数）、断言类型分布（assertEqual/assertTrue/assertRaises 比例）和预言准确率综合评估。

- **自动化级别**：🟡 半自动，AST 分析提取断言 + 变异测试验证有效性
- **工具链**：tree-sitter + 变异测试框架
- **典型值**：0.4-0.8

**推荐数据集**

- **TestGenEval** (A-)：最大规模单元测试生成 benchmark，68,647 测试，ICLR 2025
- **EvalPlus** (A)：增强测试集可作为测试充分性标杆，NeurIPS 2023

**基建需求**

- 测试执行 pipeline 搭建：约 1 周

##### C-3.2 测试覆盖能力

**评估目标**：AI 生成测试的代码覆盖深度和增量贡献。覆盖率是衡量测试充分性的经典指标，但对于 AI 生成测试，仅关注绝对覆盖率是不够的，还需评估其在已有测试基础上的增量贡献。一个优秀的 LLM 测试生成工具应当能够发现人工测试未覆盖的边界条件和异常路径，而非简单重复已有测试覆盖的代码区域。因此本框架同时引入绝对覆盖率和覆盖率增益两个互补指标。

**M-3.2.1 分支覆盖率** ↑

*branch_coverage = |覆盖的分支数| / |总分支数| × 100%*

- **自动化级别**：🟢 全自动
- **工具链**：coverage.py/pytest-cov (Python), JaCoCo (Java), istanbul (JS)
- **典型值**：50%-85%

**M-3.2.2 覆盖率增益** ↑

*Δ_coverage = coverage(existing_tests ∪ generated_tests) - coverage(existing_tests)*

衡量 AI 生成测试在已有测试基础上带来的增量覆盖率提升。

- **自动化级别**：🟢 全自动
- **工具链**：同 M-3.2.1
- **典型值**：5%-30%

**推荐数据集**

- **TestGenEval** (A-)：含覆盖率度量，11 个 Python 仓库，ICLR 2025

**基建需求**

- 覆盖率工具集成：约 0.5 人天

##### C-3.3 缺陷检测有效性

**评估目标**：测试套件发现缺陷的能力、回归安全性和执行稳定性。测试的最终目的是发现缺陷并保障回归安全。覆盖率高并不等同于缺陷检测能力强，因为覆盖了某一代码路径不意味着断言能检出该路径上的错误。变异测试通过系统性地向源代码注入人工缺陷来评估测试套件的真实检错能力，是衡量测试有效性的金标准方法。此外，回归安全性（P2P）和测试稳定性两个指标分别保障修改后的系统仍然可靠、以及评测结果本身不受不稳定测试的干扰。

**M-3.3.1 变异测试得分** ↑

*mutation_score = |被杀死的变异体数| / |非等价变异体总数| × 100%*

向源代码注入人工缺陷（算术运算符替换、关系运算符替换、语句删除等），检验测试套件能否检出这些缺陷。

- **自动化级别**：🟢 全自动
- **工具链**：mutmut (Python), PIT (Java), Stryker (JS/TS)
- **典型值**：40%-80%

**M-3.3.2 回归安全性 P2P** ↑

*P2P = |修改后仍通过的原有测试数| / |修改前通过的测试总数| × 100%*

SWE-bench 要求 P2P 全部保持（100%），这是极严格的标准。

- **自动化级别**：🟢 全自动
- **工具链**：测试框架 + diff 比较脚本
- **典型值**：90%-100%

**M-3.3.3 测试稳定性** ↑

*determinism_score = |连续 N 次运行结果完全一致的测试| / |总测试数| × 100%*    # N ≥ 5

排除不稳定测试（flaky test）对评测结果的干扰。

- **自动化级别**：🟢 全自动
- **工具链**：pytest-repeat + 结果比对脚本
- **典型值**：85%-99%

**推荐数据集**

- **EvalPlus** (A)：增强测试暴露脆弱代码，NeurIPS 2023
- **Defects4J v2.0** (A)：835 个 Java bugs，20+ 年社区验证，APR 领域金标准
- 变异测试基建 (—)：mutmut/PIT 工具直接度量

**基建需求**

- 变异测试框架配置：约 2-3 天
- 计算资源开销较大（变异体数量级大）

#### 阶段 4：维护与演进

> SWEBOK 映射：Program Comprehension, Corrective Maintenance, Perfective Maintenance

软件维护占据软件总生命周期成本的 60% 以上，是决定 LLM 工程实用价值的关键阶段。与从零生成代码不同，维护任务要求模型理解现有代码库的复杂上下文、追踪跨文件的调用链、精准定位缺陷根因并生成最小化补丁。随着 AI 编程助手从代码补全工具向自主开发 Agent 演进，维护与演进能力的评估重要性日益凸显。本框架从程序理解、缺陷定位与修复、代码审查与改进三个能力维度，系统评估模型在真实维护场景下的表现。

##### C-4.1 程序理解能力

**评估目标**：模型对代码库的跨文件追踪能力和根因分析能力。程序理解是所有维护任务的前提条件：只有准确理解了代码的调用关系和数据流向，才能有效定位缺陷和实施修复。当前 LLM 在单文件代码理解上已表现出色，但在涉及多文件依赖和复杂调用链的真实项目中，理解能力显著下降。本框架通过跨文件追踪深度和根因分析准确性两个指标，量化评估模型在项目级代码理解任务中的表现。

**M-4.1.1 跨文件追踪深度** ↑

*cross_file_depth = Σ(Bug_i 涉及的文件数) / |总 Bug 数|*

衡量模型在定位和修复 Bug 时是否追踪了跨文件的调用链。

- **自动化级别**：🟢 全自动，git diff 解析涉及的文件数
- **工具链**：git diff + 文件计数脚本
- **典型值**：1.5-5.0

**M-4.1.2 根因分析准确性** ↑

*root_cause_accuracy = |根因分析正确的 Bug 数| / |总 Bug 数| × 100%*

通过 LLM-as-Judge 对比模型输出与专家标注的根因分析。

- **自动化级别**：🟡 半自动，LLM-as-Judge 对比
- **工具链**：LLM API + 专家标注
- **典型值**：40%-75%

**推荐数据集**

- **CRUXEval** (B+)：代码推理理解，I/O 预测，ICML 2024，Meta

**基建需求**

- 根因分析标注：100 个 Bug × 2 名开发者，约 4 人周

##### C-4.2 缺陷定位与修复能力

**评估目标**：Bug 定位准确率、修复成功率、补丁质量。缺陷定位与修复是自动程序修复（APR）领域的核心任务，也是 LLM 在维护阶段最具实用价值的应用场景。本框架从三个粒度评估定位能力（文件级、函数级、行级），并通过修复成功率和补丁最小性两个维度评估修复质量。值得注意的是，实证数据显示当前 Agent 在文件级定位上 F1 可达 94.59%，但函数级骤降至 24.32%，说明定位粒度的细化仍是重要挑战。同时，生成补丁的膨胀问题（平均 47.54 行 vs 金标准 7.71 行）表明模型倾向于过度修改，补丁最小性的评估因此不可或缺。

**M-4.2.1 Bug 定位准确率** ↑

*文件级准确率 = |正确定位到目标文件的比例|*
*函数级准确率 = |正确定位到目标函数的比例|*
*行级准确率   = |正确定位到目标行的比例|*（±5 行容差）

实证数据：Agent 文件级 F1 高达 94.59%，但函数级骤降至 24.32%。

- **自动化级别**：🟢 全自动，与金标准补丁位置对比
- **工具链**：git diff 解析 + 位置匹配脚本
- **典型值**：40%-80%（行级）

**M-4.2.2 修复成功率** ↑

*fix_rate = |成功修复的 Bug 数| / |尝试修复的 Bug 数| × 100%*

成功修复：修复后所有测试通过 + 未引入新 Bug（P2P 100%）。

- **自动化级别**：🟢 全自动
- **工具链**：测试执行 + 回归验证
- **典型值**：30%-70%

**M-4.2.3 补丁最小性** ↓（膨胀比）

*patch_bloat_ratio = |生成补丁行数| / |金标准补丁行数|*

实证数据：部分 Agent 平均 47.54 行 vs 金标准 7.71 行（bloat 6.2x）。理想值接近 1.0。

- **自动化级别**：🟢 全自动
- **工具链**：git diff --stat
- **典型值**：1.0-6.0

**推荐数据集**

- **Defects4J v2.0** (A)：APR 领域金标准，835 Java bugs，~2500 引用
- **SWE-bench Multilingual** (B+)：跨语言修复能力，300 任务，9 语言，2025

**基建需求**

- 位置匹配脚本：约 2-3 天

##### C-4.3 代码审查与改进能力

**评估目标**：AI 辅助代码审查的缺陷发现能力、意见质量和跨文件一致性理解。代码审查是软件质量保障的重要实践，也是开发团队中耗时最多的协作活动之一。LLM 辅助代码审查的评估不仅需要考察其发现缺陷的精度和召回率，更需要评估审查意见的可操作性——模糊的意见对开发者而言几乎没有价值。此外，随着项目规模增大，跨文件的一致性问题（如接口签名变更后调用方未同步更新）成为人工审查容易遗漏的盲区，也是评估 LLM 审查能力的重要维度。

**M-4.3.1 缺陷 F1 得分** ↑

*Precision = |发现的真实缺陷数| / |报告的缺陷总数|*
*Recall    = |发现的真实缺陷数| / |专家标注的缺陷总数|*
*F1        = 2 × Precision × Recall / (Precision + Recall)*

- **自动化级别**：🟡 半自动，需专家标注真实缺陷作为金标准
- **工具链**：缺陷匹配脚本 + 专家标注
- **典型值**：0.45-0.75

**M-4.3.2 审查意见质量** ↑

*review_quality = Σ(意见_i 的可操作性得分) / |审查意见总数|*

5 分：具体指出问题 + 提供修复建议 + 解释原因；3 分：指出问题但缺少修复建议；1 分：模糊的意见，无法操作。

- **自动化级别**：🔴 人工，需专家评判意见质量
- **工具链**：专家评分 + LLM-as-Judge 辅助
- **典型值**：2.5-4.0

**M-4.3.3 跨文件一致性检测率** ↑

*cross_file_detection = |发现的跨文件问题数| / |标注的跨文件问题总数| × 100%*

- **自动化级别**：🟡 半自动
- **工具链**：AST 解析 + 类型检查
- **典型值**：30%-70%

**M-4.3.4 意见分类准确率** ↑

*classification_accuracy = |分类正确的意见数| / |总意见数| × 100%*

分类体系：Bug, Style, Design, Performance, Security。

- **自动化级别**：🟡 半自动
- **工具链**：LLM-as-Judge + 专家验证
- **典型值**：60%-85%

**推荐数据集**

- **CodeReviewer** (A-)：微软出品，116K+ 样本，9 语言，FSE 2022

**基建需求**

- 审查意见质量标注：150 条 × 3 标注员，约 6 人周
- 跨文件一致性标注：约 4-6 人周

### 7.2 各阶段评分公式

**阶段 1（需求与设计）**：

*S₁ = 0.25 × IFR + 0.20 × API 选择合理性 + 0.25 × (1 - LCOM) + 0.15 × 领域需求覆盖率 + 0.15 × 歧义识别率*

**阶段 2（软件构造）**：

*S₂ = 0.30 × pass@1 + 0.15 × 部分正确性 + 0.15 × (1/NET) + 0.10 × (1/NTMU) + 0.15 × API 调用正确率 + 0.15 × (1 - 漏洞引入率)*

**阶段 3（测试与验证）**：

*S₃ = 0.30 × 变异测试得分 + 0.25 × 分支覆盖率 + 0.25 × 测试通过率 + 0.20 × P2P*

**阶段 4（维护与演进）**：

*S₄ = 0.20 × Bug 定位准确率 + 0.25 × 修复成功率 + 0.15 × (1/max(补丁膨胀比,1)) + 0.25 × 缺陷 F1 + 0.15 × 审查意见质量/5*

**总分**：

*Total = Σᵢ (wᵢ × Sᵢ)*    # wᵢ 由场景化权重配置决定

| **通用评测** | 20% | 40% | 20% | 20% | 平衡各阶段能力 |

## 8. 可扩展性指南

### 8.1 四类扩展点

本框架在四个层次提供正式扩展机制：

| 扩展类型 | 说明 | 示例 |
|---------|------|------|
| **能力扩展** | 在现有阶段下新增能力维度 | 在阶段 2 新增"C-2.4 安全编码能力" |
| **指标扩展** | 在现有能力下新增指标 | 在 C-2.2 下新增"M-2.2.5 可读性" |
| **数据集扩展** | 为现有指标关联新数据集 | 为 M-2.1.1 新增 APPS 数据集 |
| **阶段扩展** | 新增生命周期阶段 | 新增"阶段 5：Agent 与交互式开发" |

**扩展流程**：

1. 按指标注册表规范（8.2 节）填写 YAML 卡片
2. 确定指标的功能角色（诊断 / 结果 / 成本）
3. 关联数据集并完成六维质量评估
4. 提交审查后合入注册表

#### Agent 指标扩展模块（预留）

Agent 与交互式开发指标作为阶段扩展预留，不纳入核心 4 阶段评分。预留能力维度：

| 能力 ID | 能力名称 | 预留指标 |
|---------|---------|---------|
| C-5.1 | 多轮交互效率 | 平均轮数、success@N、成本效率 |
| C-5.2 | 工具调用正确性 | schema 正确率、执行成功率、语义有效率 |
| C-5.3 | 跨文件编辑一致性 | 编辑冲突率、接口签名匹配率 |

### 8.2 指标注册表规范

每个指标采用 YAML 格式注册：

```yaml
metric_id: M-X.Y.Z
name: 指标名称
name_en: Metric Name (English)
capability: C-X.Y
direction: up | down          # up=越高越好, down=越低越好
formula: "公式文本"
unit: "%"                     # 取值单位
range: [0, 100]               # 取值范围
automation: auto | semi | manual
tools:
  - name: "工具名"
    url: "https://..."
datasets:
  - name: "数据集名"
    quality_rating: "A"
role: diagnostic | outcome | cost  # 指标功能角色
priority: P0 | P1 | P2
typical_range: [40, 85]
references:
  - "Author et al., Year (Venue)"
version: "3.0"
added_in: "3.0"
```

### 8.3 版本协议

- **主版本号**（X.0）：阶段结构或能力体系变更
- **次版本号**（X.Y）：新增指标或数据集
- **修订号**（X.Y.Z）：公式修正或工具链更新
- 所有指标在注册表中标注 `added_in` 版本号
- 废弃指标标注 `deprecated_in` 版本号并保留至少一个主版本周期

| 审查意见质量 | M-4.3.2 | P1 | 3 人 × 2 周 | 150 条意见，3 名开发者独立评分(1-5 分) |
| 跨文件一致性 | M-4.3.3 | P2 | 2-3 人 × 2 周 | 标注跨文件关联问题 |

### 质量控制

- **一致性检验**：Fleiss' Kappa ≥ 0.7（3+ 标注员）或 Cohen's Kappa ≥ 0.8（2 标注员）
- **冲突解决**：标注员独立完成后，对分歧案例进行讨论并达成共识
- **LLM-as-Judge 校准**：基于种子数据训练，人工抽检覆盖率 P0 ≥ 20%、P1 ≥ 10%、P2 ≥ 5%

- P1 任务总计：约 26 人周
- P2 任务总计：约 14-22 人周
