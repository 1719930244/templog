# 软件生命周期驱动的 AI 代码生成细粒度评测方案 v2.0

> **版本**: v2.0 | **日期**: 2026-03-09
> **版本说明**：合并 v1.0（框架结构 + 工具链）与 v1.1（补充指标 + 指标分层 + 场景化权重 + 人工标注方案）

## 总览

以软件生命周期的流程（需求设计、软件构造、测试验证、维护演进）进行维度划分与逻辑合并。方案在设计上重点考虑了以下三个维度：

1. **数据复用性与一致性**：通过在不同阶段复用同一数据集，追踪模型从架构设计到具体实现的一致性表现，易于实操
2. **时效性与有效性**：考虑已有 SOTA 模型在测评集的表现，保证评测结果有价值导向
3. **细粒度指标拆分**：不同于仅关注"Issue 是否修复、功能是否实现"的结果导向，本方案引入指令遵循率 (IFR)、模块内聚度 (LCOM)、变异测试得分等多种细粒度指标，深度剖析模型在软件工程各环节的"中间过程"质量

## 核心思路与划分依据

### 为什么按软件生命周期 4 阶段划分？

**传统评测的局限**：
- 现有 AI 代码生成评测（如 HumanEval, MBPP）主要关注"代码是否通过测试"（pass@k），忽略了软件工程的全生命周期
- 单一的 pass@k 指标无法区分"差一点通过"和"完全错误"，也不衡量代码质量、可维护性、安全性等非功能属性
- 真实软件开发不仅是"写代码"，还包括需求理解、架构设计、测试生成、Bug 修复、代码审查等多个阶段

**本方案的设计理念**：
1. **阶段 1（需求与设计）**：评估 AI 对需求的理解和转化能力
   - 评测重点：指令遵循、API 选择、架构设计、领域需求理解
   - 数据集：BigCodeBench + ProjDevBench + EvoCodeBench

2. **阶段 2（软件构造）**：评估生成代码的质量和效率
   - 评测重点：代码质量（异味、复杂度）、执行效率（时间、内存）、API 调用正确性
   - 数据集：ProjDevBench + EvoCodeBench + BigCodeBench

3. **阶段 3（测试）**：评估 AI 生成测试用例的能力
   - 评测重点：测试生成质量、覆盖率、变异测试得分
   - 数据集：TestGenEval + EvoCodeBench

4. **阶段 4（维护）**：评估 AI 在 Bug 修复和代码审查中的能力
   - 评测重点：Bug 定位与修复、代码审查质量
   - 数据集：RepoDebug + AACR-Bench

## 阶段 1：需求与设计

### 1.1 BigCodeBench：指令遵循与 API 设计

**典型示例**：
- 任务：实现数据分析管道，需调用 `pandas`, `numpy`, `matplotlib` 等库
- 输入：自然语言需求描述 + 139 个可用库的文档 + **约束条件**（如"不使用 for 循环"、"函数签名必须是 def analyze(data: pd.DataFrame) -> dict"）
- 预期输出：包含多个 API 调用的函数实现（平均 5-7 个函数调用/任务）

**评测维度**：
1. **指令遵循率 (IFR)**（核心指标）
   - 硬约束：AST 结构、导入限制、禁用 API（可自动验证）
   - 运行时约束：行为、性能、输出格式（需执行验证）
   - 软约束：代码风格、命名偏好（需 LLM 判定）

2. **API 选择合理性**（设计层面）
   - 是否选择了最适合需求的库（如用 `pandas.DataFrame` 而非手动循环）
   - 是否避免了冗余依赖

3. **库依赖设计**
   - 依赖数量是否合理
   - 是否存在循环依赖风险

**指标计算**：

```
指令遵循率 (IFR)：

IFR = (IFR_hard × w_hard + IFR_runtime × w_runtime + IFR_soft × w_soft) / (w_hard + w_runtime + w_soft)

IFR_hard    = 通过的硬约束数 / 硬约束总数
IFR_runtime = 通过的运行时约束数 / 运行时约束总数
IFR_soft    = 通过的软约束数 / 软约束总数

权重建议：w_hard = 3, w_runtime = 2, w_soft = 1
当某类约束为空时，该类不计入分母

验证方法：
- 硬约束：tree-sitter AST 检查（全自动）
- 运行时约束：执行验证（全自动）
- 软约束：LLM-as-Judge（半自动）

API 选择合理性得分 = (正确 API 数 / 总 API 调用数) × 100%
- 正确 API：专家标注的最优 API 选择

依赖复杂度 = Σ(库 i 的依赖深度) / 使用的库总数
- 依赖深度：该库在依赖树中的层级（直接依赖=1，间接依赖=2，...）
- 理想值 ≤ 1.5
```

### 1.2 ProjDevBench：架构设计与模块划分

**典型示例**：
- 仓库：`flask-restful` 项目（RESTful API 框架）
- 需求：设计一个支持资源路由、请求解析、响应序列化的架构
- 预期输出：多文件项目结构（5-10 个模块文件）

**评测维度**：
1. **模块划分合理性**：是否遵循单一职责原则、模块间耦合度
2. **设计异味检测**：Cyclic Dependency、God Class、Feature Envy

**指标计算**：

```
模块内聚度 (LCOM) = 1 - (Σ 方法对共享属性数 / (方法数 × 属性数))
- 理想值：LCOM ≤ 0.3（高内聚）

模块耦合度 (CBO) = 该模块引用的外部类数 + 引用该模块的外部类数
- 理想值：CBO ≤ 5

设计异味得分 = 1 - (检测到的设计异味数 / 总模块数)
- 使用 Designite 工具检测
```

### 1.3 EvoCodeBench：领域需求理解

**典型示例**：
- 领域：Database（数据库操作）
- 任务：实现一个支持事务、索引、查询优化的 ORM 层
- 10 个领域：Database, Internet, GUI, Algorithm, Data Science, Security, System, Testing, DevOps, Multimedia

**指标计算**：

```
领域需求覆盖率 = (实现的领域核心功能数 / 领域核心功能总数) × 100%
- 例如 Database 领域：CRUD, Transaction, Index, Query Optimization

领域特定改进 (DSI) = (模型在该领域的 Pass@1 - 模型在所有领域的平均 Pass@1) / 模型在所有领域的平均 Pass@1
- DSI > 0：模型的"舒适区"；DSI < 0：模型的"陌生区"
```

### 1.4 需求歧义处理能力（人工标注指标）

**定义**：当需求描述存在歧义、不完整或矛盾时，模型是否能识别并主动澄清，而非盲目生成。

**评测维度**：
1. **歧义识别率**：模型能识别出多少歧义需求
2. **澄清质量**：提出的澄清问题是否切中要害
3. **容错能力**：在无法澄清时，是否给出合理的默认假设

**指标计算**：

```
歧义识别率 = (识别出的歧义需求数 / 总歧义需求数) × 100%

澄清质量得分 = Σ(澄清问题 i 的质量评分) / 澄清问题总数
- 质量评分：专家评分（1-5 分）
```

**实现路径**：LLM-as-Judge 初筛 + 人工终审，人力需求 2-3 人×2-4 周

## 阶段 2：软件构造

### 2.1 ProjDevBench：代码质量与实现异味

**典型示例**（复用阶段 1 的 `flask-restful` 项目）：
- 架构设计已完成（阶段 1 输出），评估具体实现的代码质量
- 生成代码量级：500-1000 行 Python 代码，分布在 5-10 个文件

**评测维度**：
1. **细粒度代码属性**：可读性（命名规范、注释质量）、可维护性（函数长度、圈复杂度）、健壮性（异常处理、边界条件）
2. **实现异味检测**：Long Method、Magic Number、Duplicate Code

**指标计算**：

```
圈复杂度 (CC) = E - N + 2P
- E：控制流图的边数，N：节点数，P：连通分量数
- 理想值：CC ≤ 10

代码重复率 = (重复代码行数 / 总代码行数) × 100%
- 重复代码：连续 6 行以上的相同代码块，理想值 ≤ 5%

实现异味密度 = 检测到的实现异味数 / KLOC
- 使用 Pylint/Flake8 检测，理想值 ≤ 5 个/KLOC
```

### 2.2 EvoCodeBench：代码效率与资源消耗

**典型示例**（复用阶段 1 的 Database 领域任务）：
- 任务：实现 B+ 树索引的插入和查询
- 评估：执行时间、内存消耗

**指标计算**：

```
时间效率得分 = 1 - (实际执行时间 / 超时阈值)
- 得分范围 [0, 1]

内存效率得分 = 1 - (峰值内存消耗 / 内存限制)
- 得分范围 [0, 1]

执行效率归一化 (NET) = 生成代码执行时间 / 基准解执行时间
- NET = 1：与基准解相同；NET > 1：劣于基准解
- 执行效率得分 = 1 / NET

算法复杂度匹配度 = (实际复杂度 == 理论最优复杂度) ? 1 : 0
```

### 2.3 BigCodeBench：API 调用正确性

**典型示例**（复用阶段 1 的数据分析任务）：
- API 调用序列：`pd.read_csv()` → `df.groupby()` → `df.agg()` → `plt.plot()`

**指标计算**：

```
API 调用正确率 = (正确的 API 调用数 / 总 API 调用数) × 100%
- 正确的 API 调用：参数类型正确 + 返回值正确处理 + 无运行时错误

异常处理覆盖率 = (有异常处理的 API 调用数 / 可能抛出异常的 API 调用数) × 100%
```

### 2.4 部分正确性（Partial Correctness）

**定义**：当代码无法完全通过所有测试用例时，衡量其通过的测试用例比例。

```
部分正确性 = (通过的测试用例数 / 总测试用例数) × 100%
- 与 pass@k 的区别：pass@k 为二值判定，部分正确性为连续值
```

## 阶段 3：测试

### 3.1 TestGenEval：单元测试生成质量

**数据规模**：68,647 个测试用例，来自 11 个 Python 仓库

**指标计算**：

```
测试通过率 = (通过的测试用例数 / 生成的测试用例总数) × 100%
- 通过的测试：语法正确 + 运行无错误 + 断言有效

分支覆盖率 = (覆盖的分支数 / 总分支数) × 100%
- 使用 coverage.py 工具测量

变异测试得分 = (杀死的变异体数 / 总变异体数) × 100%
- 变异体：对源代码进行微小修改（如 + 改为 -）后的版本
- 使用 mutmut 工具生成变异体
```

### 3.2 EvoCodeBench：测试覆盖率评估

**指标计算**：

```
语句覆盖率 = (执行的语句数 / 总语句数) × 100%

路径覆盖率 = (执行的路径数 / 总路径数) × 100%
- 实际中通常采样关键路径
```

## 阶段 4：维护

### 4.1 RepoDebug：Bug 定位与修复

**数据规模**：8 种编程语言，22 种错误类型

**评测维度**：
1. **Bug 定位准确性**：是否定位到正确的文件和行号，是否识别了根本原因
2. **修复策略合理性**：修复是否解决了根本问题，是否引入了新 Bug
3. **跨文件追踪能力**：是否追踪了跨文件的调用链

**指标计算**：

```
Bug 定位准确率 = (正确定位的 Bug 数 / 总 Bug 数) × 100%
- 正确定位：定位到的文件和行号在专家标注的范围内（±5 行）

修复成功率 = (成功修复的 Bug 数 / 尝试修复的 Bug 数) × 100%
- 成功修复：修复后所有测试通过 + 未引入新 Bug

跨文件追踪深度 = Σ(Bug i 涉及的文件数) / 总 Bug 数

根因分析准确性 = (根因分析正确的 Bug 数 / 总 Bug 数) × 100%
- 评估方法：LLM-as-Judge 对比模型输出与专家标注

补丁最小性比值 = 生成补丁行数 / 金标准补丁行数
- 补丁最小性得分 = 1 / max(补丁最小性比值, 1)
```

### 4.2 AACR-Bench：代码审查质量（含人工标注）

**数据规模**：AI 辅助 + 专家验证，缺陷发现数比传统方法多 285%

**评测维度**：
1. **缺陷发现能力**：是否发现了所有专家标注的缺陷
2. **审查质量**（需人工评估）：审查意见是否具体、可操作，意见分类是否准确
3. **跨文件上下文理解**：是否理解了代码变更对其他模块的影响

**指标计算**：

```
缺陷召回率 = (发现的真实缺陷数 / 专家标注的缺陷总数) × 100%
缺陷精确率 = (发现的真实缺陷数 / 报告的缺陷总数) × 100%
F1 得分 = 2 × (精确率 × 召回率) / (精确率 + 召回率)

审查意见质量 = Σ(意见 i 的可操作性得分) / 审查意见总数
- 5 分：具体指出问题 + 提供修复建议 + 解释原因
- 3 分：指出问题但缺少修复建议
- 1 分：模糊的意见，无法操作

意见分类准确率 = (分类正确的意见数 / 总意见数) × 100%
- 分类：Bug, Style, Design, Performance, Security

跨文件一致性检测率 = (发现的跨文件问题数 / 标注的跨文件问题总数) × 100%
```

| 生成成本（$/任务） | 0-∞ | ↓ | $0.01-$0.50 | 🟢 全自动 |

| 响应时间（秒） | 0-∞ | ↓ | 1-30 秒 | 🟢 全自动 |
| 开发者采纳率 | 0-100% | ↑ | 20-60% | 🔴 人工 |

**自动化程度说明**：🟢 全自动 | 🟡 半自动（需专家标注或验证） | 🔴 人工（需专家全程参与）

## 各阶段综合得分计算

**阶段 1（需求与设计）**：
```
需求设计得分 = 0.25 × IFR + 0.20 × API 选择合理性 + 0.25 × (1 - LCOM) + 0.15 × 领域需求覆盖率 + 0.15 × 歧义识别率
```

**阶段 2（构造）**：
```
构造得分 = 0.30 × (1 - 实现异味密度/10) + 0.30 × 执行效率得分 + 0.20 × 内存效率得分 + 0.20 × API 调用正确率
```

**阶段 3（测试）**：
```
测试得分 = 0.40 × 变异测试得分 + 0.30 × 分支覆盖率 + 0.30 × 测试通过率
```

**阶段 4（维护）**：
```
维护得分 = 0.25 × Bug 定位准确率 + 0.25 × 修复成功率 + 0.15 × 补丁最小性得分 + 0.20 × 缺陷 F1 得分 + 0.15 × 审查意见质量
```

| 评测场景 | 需求与设计 | 软件构造 | 测试 | 维护 | 说明 |
|---------|-----------|---------|------|------|------|

总分 = Σ (阶段权重 × 阶段内结果指标加权分)

## 评测流程

```
1. 数据准备
   ├─ 加载数据集（BigCodeBench, ProjDevBench 等）
   ├─ 解析任务描述（需求、约束、测试用例）
   └─ 准备评测环境（依赖安装、沙箱配置）

2. 代码生成
   ├─ 模型推理（Complete/Instruct 模式）
   ├─ 代码清洗（去除注释、格式化）
   └─ 语法检查（tree-sitter 验证）

3. 阶段 1 评测（需求与设计）
   ├─ IFR 计算（硬约束 + 运行时约束 + 软约束）
   ├─ API 选择分析（专家标注对比）
   ├─ 架构质量（LCOM, CBO, 设计异味）
   └─ 领域需求覆盖（功能清单对比）

4. 阶段 2 评测（软件构造）
   ├─ 代码质量（CC, 重复率, 实现异味）
   ├─ 执行效率（时间、内存监控）
   └─ API 正确性（单元测试 + 异常处理）

5. 阶段 3 评测（测试）
   ├─ 测试生成（pytest 执行）
   ├─ 覆盖率分析（语句、分支、路径）
   └─ 变异测试（mutmut 变异体生成）

6. 阶段 4 评测（维护）
   ├─ Bug 定位（专家标注对比）
   ├─ 修复验证（测试通过 + 回归测试 + 补丁最小性）
   └─ 代码审查（缺陷发现 + 意见质量 + 跨文件一致性）

7. 结果汇总
   ├─ 各阶段得分计算
   ├─ 综合得分计算（按场景化权重）
   └─ 生成评测报告（JSON + 可视化）
```

## 人工标注数据构建方案

### API 选择合理性标注

- 从 BigCodeBench 抽样 200 个任务，3 名资深开发者独立标注最优 API 选择
- 一致性检验 Fleiss' Kappa ≥ 0.7，冲突讨论解决
- 基于种子数据训练 LLM-as-Judge 扩展到全量
- **人力**：3 人×2 周 = 6 人周

### 领域需求覆盖标注

- 每个领域招募 1-2 名专家，定义核心功能清单（5-10 个）
- 为每个功能提供正例和反例，交叉验证完整性
- **人力**：10 人×1 周 = 10 人周

### Bug 定位专家标注

- 从 RepoDebug 抽样 100 个 Bug，2 名开发者独立定位
- Cohen's Kappa ≥ 0.8，标注格式：文件路径 + 函数名 + 行号范围（±5 行）
- **人力**：2 人×2 周 = 4 人周

### 代码审查意见质量标注

- 从 AACR-Bench 抽样 150 条审查意见，3 名开发者独立评分（1-5 分）+ 意见分类
- Fleiss' Kappa ≥ 0.7，基于种子数据训练 LLM-as-Judge
- **人力**：3 人×2 周 = 6 人周

### 人工标注汇总

| 指标 | 阶段 | 人力需求 | 优先级 |
|------|------|----------|--------|
| API 选择合理性 | 需求与设计 | 6 人周 | P1 |
| 领域需求覆盖 | 需求与设计 | 10 人周 | P1 |
| 需求歧义处理 | 需求与设计 | 6-12 人周 | P2 |
| Bug 定位准确性 | 维护 | 4 人周 | P1 |
| 根因分析准确性 | 维护 | 4 人周 | P2 |
| 审查意见质量 | 维护 | 6 人周 | P1 |
| 跨文件一致性检测 | 维护 | 4-6 人周 | P2 |

**P1 总计**：26 人周 | **P2 总计**：14-22 人周
